このページでは2つの統合段階について説明します：

1. **直接統合** — CDNエッジがオリジンサーバーに直接転送（ベースラインテスト）
2. **F5 XC挿入** — CDNとオリジン間にF5 XC HTTPロードバランサーを挿入（セキュリティデモ）

まず直接統合でベースラインを確立し、準備ができたらF5 XCを挿入します。

## オリジンサーバーリファレンス

[origin-server](https://f5xc-salesdemos.github.io/origin-server/)ラボコンポーネントは、セキュリティテスト用の脆弱なWebアプリケーションを提供します。

| プロパティ | 値 |
|---|---|
| ドキュメント | [f5xc-salesdemos.github.io/origin-server](https://f5xc-salesdemos.github.io/origin-server/) |
| リポジトリ | [github.com/f5xc-salesdemos/origin-server](https://github.com/f5xc-salesdemos/origin-server) |
| デフォルトポート | 80 |
| ヘルスチェック | `GET /health` |

### 利用可能なアプリケーション

オリジンサーバーの公開マニフェストから現在のアプリケーションカタログを取得します：

```bash
MANIFEST_URL=$(curl -sf https://api.github.com/repos/f5xc-salesdemos/origin-server/releases/latest \
  | python3 -c "import sys,json; assets=json.load(sys.stdin).get('assets',[]); print(next((a['browser_download_url'] for a in assets if a['name']=='manifest.json'),''))")
curl -sf "$MANIFEST_URL" | python3 -m json.tool
```

リリースがまだ存在しない場合は、リポジトリソースを直接使用します：

```bash
curl -sf https://raw.githubusercontent.com/f5xc-salesdemos/origin-server/main/manifest.json | python3 -m json.tool
```

マニフェストには、すべてのアプリケーションパス、ヘルスチェック、コンテナイメージ、デモ機能マッピングが記載されています。

### アプリケーションパス

## ステージ1：直接統合（ベースライン）

```
┌──────────┐     ┌──────────────────────┐     ┌─────────────────────┐
│  クライアント │────▶│  CDNエッジ (NGINX)    │────▶│  オリジンサーバー     │
│           │     │  20.65.90.112        │     │  20.12.78.159       │
└──────────┘     │  67以上のCDNヘッダー   │     │  Juice Shop, DVWA,  │
                 │  ディスクキャッシュ      │     │  VAmPI, httpbin,    │
                 └──────────────────────┘     │  whoami, CSD Demo   │
                                              └─────────────────────┘
```

### オリジンの設定

CDNエッジのNGINX設定でオリジンサーバーのIPを設定します：

```bash
ssh azureuser@<CDN_EDGE_IP>
sudo sed -i 's|proxy_pass .*;|proxy_pass http://<ORIGIN_IP>;|' /etc/nginx/conf.d/cdn-edge.conf
sudo rm -rf /var/cache/nginx/cdn/*
sudo nginx -t && sudo systemctl reload nginx
```

または、`terraform.tfvars`でTerraformデプロイ時に設定します：

```hcl
origin_server = "http://<ORIGIN_IP>"
```

### すべてのアプリケーションの検証

CDN経由で各オリジンアプリケーションをテストします：

```bash
CDN=<CDN_EDGE_IP>

# ヘルスチェック（CDNローカル）
curl -sf "http://$CDN/health" | python3 -m json.tool

# ランディングページ
curl -sf -o /dev/null -w "/ : HTTP %{http_code}\n" "http://$CDN/"

# Juice Shop
curl -sf -o /dev/null -w "/juice-shop/ : HTTP %{http_code}\n" "http://$CDN/juice-shop/"

# DVWA（ログインへリダイレクト）
curl -sf -o /dev/null -w "/dvwa/ : HTTP %{http_code}\n" -L "http://$CDN/dvwa/"

# VAmPI API
curl -sf "http://$CDN/vampi/users/v1" | python3 -m json.tool | head -5

# httpbin ヘッダー（CDNヘッダーをJSONで表示）
curl -sf "http://$CDN/httpbin/headers" | python3 -m json.tool | head -10

# whoami（オリジンが受信するすべてのヘッダーを表示）
curl -sf "http://$CDN/whoami/"

# CSD Demo
curl -sf -o /dev/null -w "/csd-demo/ : HTTP %{http_code}\n" "http://$CDN/csd-demo/"
```

すべてのパスがHTTP 200を返す必要があります（DVWAはリダイレクト後に302、次に200を返します）。

### オリジンでのCDNヘッダーの検証

`/whoami/`エンドポイントは、オリジンが受信するすべてのヘッダーを表示します。CDN経由でアクセスすると、67以上のベンダーヘッダーがすべて表示されます：

```bash
curl -sf "http://$CDN/whoami/"
```

以下の主要ヘッダーが存在することを確認します：

| ベンダー | ヘッダー | 期待される値 |
|---|---|---|
| 標準 | `X-Forwarded-For` | `<your_ip>, <cdn_edge_ip>` |
| Akamai | `True-Client-Ip` | `<your_ip>` |
| Cloudflare | `Cf-Connecting-Ip` | `<your_ip>` |
| CloudFront | `Cloudfront-Viewer-Country` | `US` |
| Fastly | `Fastly-Client-Ip` | `<your_ip>` |
| Azure FD | `X-Azure-Clientip` | `<your_ip>` |

### ログの相互参照

両方のサーバーのアクセスログを比較してトラフィックフローを検証します：

```bash
# CDNエッジログ — ソースとしてクライアントIPを表示
ssh azureuser@<CDN_EDGE_IP> "sudo tail -5 /var/log/nginx/access.log"

# オリジンログ — ソースとしてCDNエッジIPを表示
ssh azureuser@<ORIGIN_IP> "sudo tail -5 /var/log/nginx/access.log"
```

オリジンログではCDNエッジIPが接続クライアントとして表示され、実際のクライアントIPは`X-Forwarded-For`およびベンダー固有のヘッダーで伝達されます。

### キャッシュ動作

```bash
# 最初のリクエスト — MISS（オリジンから取得）
curl -s -I "http://$CDN/whoami/" | grep X-Cache-Status

# 2回目のリクエスト — HIT（CDNキャッシュから配信）
curl -s -I "http://$CDN/whoami/" | grep X-Cache-Status
```

## ステージ2：F5 XC挿入（セキュリティデモ）

ベースラインテスト後、CDNとオリジン間にF5 XC HTTPロードバランサーを挿入します：

```
┌──────────┐     ┌────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  クライアント │────▶│  CDNエッジ      │────▶│  F5 XC HTTP LB   │────▶│  オリジンサーバー │
│           │     │  (NGINX)       │     │  WAF + Bot + API  │     │                 │
└──────────┘     └────────────────┘     └──────────────────┘     └─────────────────┘
```

1. **F5 XC HTTPロードバランサーを作成**し、オリジンプールにオリジンサーバーを設定します
2. **CDNエッジを更新**して、オリジンに直接ではなくF5 XC VIPを指すようにします：

```bash
ssh azureuser@<CDN_EDGE_IP>
sudo sed -i 's|proxy_pass .*;|proxy_pass https://<F5XC_LB_VIP>;|' /etc/nginx/conf.d/cdn-edge.conf
sudo rm -rf /var/cache/nginx/cdn/*
sudo nginx -t && sudo systemctl reload nginx
```

3. **WAF適用を検証**して、フルチェーンを通じて確認します：

```bash
# CDN経由のSQLインジェクション → F5 XC WAFがブロックするはず
curl -I "http://$CDN/dvwa/vulnerabilities/sqli/?id=%27+OR+1%3D1--"

# 通常のリクエストは通過するはず
curl -sf -o /dev/null -w "%{http_code}" "http://$CDN/juice-shop/"
```

4. **F5 XCで信頼済みクライアントIPヘッダーを設定**し、CDNヘッダーから実際のクライアントIPを読み取るようにします（例：Akamaiシミュレーション用の`True-Client-IP`、Cloudflareシミュレーション用の`CF-Connecting-IP`）

## マルチコンポーネントアーキテクチャ

| コンポーネント | リポジトリ | 目的 |
|---|---|---|
| **CDNエッジ**（これ） | [cdn-simulator](https://github.com/f5xc-salesdemos/cdn-simulator) | キャッシュ、ベンダーヘッダー |
| **オリジンサーバー** | [origin-server](https://github.com/f5xc-salesdemos/origin-server) | 脆弱なWebアプリ |
| **F5 XC設定** | Various (waf, api-protection, bot-*, etc.) | セキュリティポリシー |

各コンポーネントは、AIアシスタントがインフラストラクチャをデプロイするために読み取るドキュメントを公開しています。origin-serverは、すべてのアプリケーションパスとヘルスチェックを一覧にした[エンドポイントマニフェスト](https://github.com/f5xc-salesdemos/origin-server/releases/latest)をGitHub Releaseアーティファクトとして公開しています。
