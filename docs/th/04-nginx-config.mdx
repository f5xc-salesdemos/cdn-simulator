การจัดเตรียม (provisioning) ด้วย cloud-init จะดีพลอยการกำหนดค่า NGINX CDN edge ที่ปรับแต่งประสิทธิภาพอย่างสมบูรณ์ หน้านี้จะอธิบายทุกชั้นของการกำหนดค่า ตั้งแต่การปรับแต่งเคอร์เนลไปจนถึงพฤติกรรมแคชและการแทรก vendor header ทุกการตั้งค่าได้รับการตรวจสอบภายใต้การทดสอบโหลดต่อเนื่อง 48 ชั่วโมง โดยมีค่าสูงสุดที่ 172,540 req/s

## ไฟล์การกำหนดค่า

| ไฟล์ | วัตถุประสงค์ |
|---|---|
| `/etc/sysctl.d/99-cdn-tuning.conf` | การปรับแต่งเครือข่ายเคอร์เนล Linux |
| `/etc/systemd/system/nginx.service.d/override.conf` | ขีดจำกัด file descriptor สำหรับ NGINX |
| `/etc/security/limits.d/99-nginx.conf` | ขีดจำกัดระดับ OS สำหรับผู้ใช้ www-data |
| `/etc/nginx/nginx.conf` | การกำหนดค่าหลักของ NGINX (workers, buffers, gzip, logging) |
| `/etc/nginx/conf.d/cdn-edge.conf` | การกำหนดค่า CDN proxy (cache, upstream, headers) |

## การกำหนดค่าหลักของ NGINX

### Workers และ Connections

```nginx
worker_processes auto;           # 4 workers on D4s_v5 (1 per vCPU)
worker_rlimit_nofile 65535;      # Per-worker file descriptor limit

events {
    use epoll;                   # Linux-optimized event model
    worker_connections 8192;     # 4 workers x 8192 = 32,768 max concurrent
    multi_accept on;             # Accept all pending connections per event loop
    accept_mutex off;            # Not needed with epoll + reuseport
}
```

Systemd override ที่ `/etc/systemd/system/nginx.service.d/override.conf` ตั้งค่า `LimitNOFILE=65535` ให้ตรงกัน

### Proxy Buffers

```nginx
proxy_buffering on;
proxy_buffer_size 16k;           # Header buffer (handles large CDN headers)
proxy_buffers 64 16k;            # 1 MB per connection (64 x 16k)
proxy_busy_buffers_size 256k;    # Can send 256k to client while still reading
```

ขนาด busy buffer 256k เป็นสิ่งสำคัญ — จะต้องมากกว่า response ที่แคชไว้ที่ใหญ่ที่สุด (Juice Shop มีขนาด 75 KB) การตั้งค่าเดิมที่ 64k ทำให้เกิด serialization ภายใต้โหลดสูง

### การบีบอัด Gzip

```nginx
gzip on;
gzip_comp_level 4;               # Balance: ~85% of max compression at ~40% CPU
gzip_min_length 256;             # Skip tiny responses (gzip header overhead)
gzip_vary on;                    # Vary: Accept-Encoding for correct caching
gzip_proxied any;                # Compress all proxied responses
gzip_types text/plain text/css text/javascript text/xml
           application/json application/javascript application/xml
           application/xml+rss application/atom+xml
           application/ld+json application/manifest+json
           image/svg+xml;
```

### Open File Cache

```nginx
open_file_cache max=200000 inactive=20s;
open_file_cache_valid 30s;
open_file_cache_min_uses 2;
open_file_cache_errors on;
```

แคช file descriptors และ metadata สำหรับออบเจ็กต์ที่แคชไว้ ช่วยลดการเรียก syscall `stat()` และ `open()` บนไฟล์ที่ถูกเข้าถึงบ่อย

### Client Keepalive

```nginx
keepalive_timeout 65;
keepalive_requests 100000;       # Was 1000 — caused connection recycling at 90K req/s
```

ที่ 90K req/s ด้วย `keepalive_requests 1000` การเชื่อมต่อจะถูกรีไซเคิลทุก 11 วินาที ทำให้เกิดซ็อกเก็ต TIME_WAIT การเพิ่มเป็น 100,000 ช่วยแก้ปัญหานี้ได้ทั้งหมด

### การบันทึก Access Log

```nginx
log_format cdn '$remote_addr [$time_local] "$request" $status $body_bytes_sent $upstream_cache_status $request_time';
access_log /var/log/nginx/access.log cdn buffer=256k flush=5s;
```

รูปแบบ log `cdn` รวม `$upstream_cache_status` (HIT/MISS) และ `$request_time` สำหรับการวิเคราะห์ latency การบันทึกแบบบัฟเฟอร์ (256k/flush ทุก 5 วินาที) ช่วยลดภาระ I/O ภายใต้โหลดสูง

## Upstream Keepalive

```nginx
upstream origin_backend {
    server 20.12.78.159:80;
    keepalive 256;               # Persistent connections per worker to origin
    keepalive_timeout 60s;
    keepalive_requests 1000;
}
```

ด้วย 4 workers x 256 keepalive = 1,024 การเชื่อมต่อที่พร้อมใช้งานไปยัง origin เมื่อรวมกับ `proxy_http_version 1.1` และ `proxy_set_header Connection ""` จะช่วยลดภาระ TCP handshake ในทุก cache miss

## การกำหนดค่าแคช

### Cache Path

```nginx
proxy_cache_path /var/cache/nginx/cdn
                 levels=1:2
                 keys_zone=cdn_cache:32m
                 max_size=25g
                 inactive=24h
                 use_temp_path=off;
```

| พารามิเตอร์ | ค่า | วัตถุประสงค์ |
|---|---|---|
| `keys_zone=cdn_cache:32m` | 32 MB shared memory | จัดเก็บ cache keys และ metadata ประมาณ 256,000 รายการ |
| `max_size=25g` | ขีดจำกัดดิสก์ 25 GB | ใช้พื้นที่ส่วนใหญ่ของดิสก์ OS ขนาด 30 GB; การลบแบบ LRU เมื่อถึงขีดจำกัด |
| `inactive=24h` | ไทม์เอาต์ไม่ใช้งาน 24 ชั่วโมง | เนื้อหาที่ไม่ถูกเข้าถึงเป็นเวลา 24 ชั่วโมงจะถูกลบ (ไม่เหมือนกับ TTL) |
| `use_temp_path=off` | เขียนตรงไปยังไดเรกทอรีแคช | หลีกเลี่ยงการย้ายไฟล์ข้ามระบบไฟล์ |

### พฤติกรรมแคช

```nginx
proxy_cache cdn_cache;
proxy_cache_valid 200 301 302 4h;
proxy_cache_valid 404 1m;
proxy_cache_key "$scheme$host$request_uri";
proxy_cache_lock on;
proxy_cache_lock_age 3s;
proxy_cache_lock_timeout 3s;
proxy_cache_background_update on;
proxy_cache_use_stale updating error timeout http_500 http_502 http_503 http_504;
proxy_ignore_headers Set-Cookie Cache-Control Expires Vary;

proxy_hide_header X-Cache-Status;
proxy_hide_header Vary;
```

| ไดเรกทีฟ | ค่า | วัตถุประสงค์ |
|---|---|---|
| `proxy_cache_valid 200 301 302 4h` | TTL 4 ชั่วโมง | เนื้อหาที่แคชมีอายุ 4 ชั่วโมง (เพิ่มขึ้นเพื่อลดการตกของ throughput จากคลื่นการรีเฟรช) |
| `proxy_cache_lock on` | ป้องกัน thundering herd | คำขอเพียง 1 รายการเท่านั้นที่ส่งไปยัง origin ต่อ URL ที่ยังไม่ได้แคช; คำขออื่นๆ รอการเติมแคช |
| `proxy_cache_lock_age 3s` | ไทม์เอาต์ล็อก | หากคำขอแรกยังไม่เสร็จภายใน 3 วินาที อนุญาตให้อีกคำขอหนึ่งผ่านไปได้ |
| `proxy_cache_background_update on` | การรีเฟรชแบบ zero-latency | ส่งเนื้อหาเก่าทันทีขณะรีเฟรชในเบื้องหลัง |
| `proxy_cache_use_stale` | ความทนทาน | ส่งเนื้อหาเก่าระหว่างที่ origin มีข้อผิดพลาด (500/502/503/504) หรือขณะกำลังอัปเดต |
| `proxy_ignore_headers` | บังคับแคช | ละเว้น Set-Cookie, Cache-Control, Expires, Vary จาก origin — CDN เป็นผู้กำหนดพฤติกรรม TTL และ Vary (Juice Shop ส่ง `max-age=0` และ header `Vary` สามตัวซึ่งป้องกันการแคชที่มีประสิทธิภาพ) |
| `proxy_hide_header X-Cache-Status` | ลบ header ของ origin | NGINX ต้นทางเพิ่ม X-Cache-Status ของตัวเอง — ลบออกเพื่อให้เห็นเฉพาะสถานะแคชของ CDN เท่านั้น |
| `proxy_hide_header Vary` | ป้องกันการแตกกระจายของแคช | Origin ส่ง header `Vary: Accept-Encoding` หลายตัว การลบออกจะป้องกันการแตกกระจายของ cache key จากการเรียงสับเปลี่ยน `Accept-Encoding` `gzip_vary on` ของ NGINX จะเพิ่ม header `Vary` ที่ถูกต้องตัวเดียวโดยอัตโนมัติ |

### ไทม์เอาต์ Proxy

```nginx
proxy_read_timeout 30s;
proxy_connect_timeout 10s;
proxy_send_timeout 15s;
```

ให้เวลา origin ในการตอบสนองมากขึ้นภายใต้โหลดสูง ขณะเดียวกันยังคง fail fast ในกรณีที่มีปัญหาการเชื่อมต่อ

### Server Block

```nginx
server {
    listen 80 reuseport;        # Kernel distributes connections across all 4 workers
    server_name _;
}
```

`reuseport` เปิดใช้ `SO_REUSEPORT` — เคอร์เนลกระจายการเชื่อมต่อขาเข้าโดยตรงไปยัง worker processes ช่วยลดการแย่งชิง accept mutex

## CDN Vendor Headers

ตัวจำลองจะแทรก headers จาก CDN vendor หลักทั้ง 5 รายพร้อมกัน สิ่งนี้ช่วยให้สามารถกำหนดค่า F5 XC ด้วย "Trusted Client IP Header" ของ vendor ใดก็ได้ และเห็น header payloads ที่สมจริง ไม่ว่าจะจำลอง CDN ใดอยู่

### Headers มาตรฐานอุตสาหกรรม (CDN ทั้งหมด)

| Header | ค่า | วัตถุประสงค์ |
|---|---|---|
| `X-Forwarded-For` | ลำดับ IP ของไคลเอนต์ | IP ที่ส่งต่อแบบมาตรฐาน |
| `X-Forwarded-Proto` | `http` หรือ `https` | โปรโตคอลเดิมของไคลเอนต์ |
| `X-Forwarded-Host` | ชื่อโฮสต์เดิม | header Host เดิม |
| `X-Forwarded-Port` | พอร์ตของเซิร์ฟเวอร์ | พอร์ตเดิม |
| `X-Real-IP` | IP ของไคลเอนต์ | IP ของไคลเอนต์ตัวเดียว (ธรรมเนียม nginx) |
| `Via` | `1.1 cdn-simulator` | การระบุตัวตนของ proxy |
| `Forwarded` | รูปแบบ RFC 7239 | header การส่งต่อแบบมาตรฐาน |
| `CDN-Loop` | `cdn-simulator` | การตรวจจับลูป |

## การตรวจจับอุปกรณ์

ตัวจำลองตรวจจับประเภทอุปกรณ์จาก header `User-Agent` โดยใช้ไดเรกทีฟ `map` ของ NGINX:

- **มือถือ**: จับคู่ iPhone, Android (ไม่ใช่แท็บเล็ต), iPod, BlackBerry, Opera Mini, IEMobile
- **แท็บเล็ต**: จับคู่ iPad, แท็บเล็ต Android, Kindle, PlayBook
- **เดสก์ท็อป**: ค่าเริ่มต้นเมื่อไม่ตรงกับทั้งมือถือและแท็บเล็ต

ประเภทอุปกรณ์จะสะท้อนใน:
- `CloudFront-Is-Desktop-Viewer` / `CloudFront-Is-Mobile-Viewer` / `CloudFront-Is-Tablet-Viewer`
- `X-Akamai-Device-Characteristics` (ฟิลด์ `is_mobile`, `is_tablet`, `is_wireless_device`)

## การดำเนินงาน

### การเปลี่ยนเซิร์ฟเวอร์ Origin

```bash
ssh azureuser@<PUBLIC_IP>

# Update upstream server
sudo sed -i 's|server .*;|server NEW_HOST:80;|' /etc/nginx/conf.d/cdn-edge.conf

# Clear cache and reload
sudo rm -rf /var/cache/nginx/cdn/*
sudo nginx -t && sudo systemctl reload nginx
```

หรืออัปเดต `origin_host` ใน `terraform.tfvars` แล้วรัน `terraform apply` เพื่อจัดเตรียมใหม่

### การล้างแคช

```bash
ssh azureuser@<PUBLIC_IP>
sudo rm -rf /var/cache/nginx/cdn/*
sudo systemctl reload nginx
```

### การตรวจสอบสถิติแคช

```bash
# Object count and disk usage
sudo find /var/cache/nginx/cdn -type f | wc -l
sudo du -sh /var/cache/nginx/cdn

# Recent access log with cache status
tail -20 /var/log/nginx/access.log
```

### การตรวจสอบภายใต้โหลด

```bash
# Real-time connections and socket states
ss -s

# NGINX worker CPU usage
top -bn1 | grep nginx

# Upstream keepalive connections
ss -tn state established dst <ORIGIN_IP> | wc -l

# TIME_WAIT socket count
ss -tn state time-wait | wc -l
```
