หลังจากการ deploy แล้ว ให้รันการตรวจสอบเหล่านี้เพื่อยืนยันว่า CDN edge node ทำงานได้อย่างถูกต้อง ให้รอ 2-3 นาทีหลังจาก `terraform apply` เพื่อให้ cloud-init ติดตั้ง NGINX เสร็จสมบูรณ์

## การตรวจสอบสุขภาพ

ตรวจสอบว่า NGINX กำลังทำงานและตอบสนองอยู่:

```bash
curl -s "http://<PUBLIC_IP>/health" | jq .
```

ผลลัพธ์ที่คาดหวัง:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

หากใช้ Terraform outputs:

```bash
curl -s "$(terraform output -raw health_check_url)" | jq .
```

## Cache Miss (คำขอแรก)

คำขอแรกไปยัง path ใดๆ จะเป็น cache miss — edge จะดึงข้อมูลจาก origin:

```bash
curl -I "http://<PUBLIC_IP>/"
```

มองหา headers เหล่านี้ในการตอบสนอง:

```
X-Cache-Status: MISS
X-CDN-Edge: cdn-simulator
```

`MISS` หมายความว่าเนื้อหาไม่อยู่ในแคชและถูกดึงมาจาก origin server

## Cache Hit (คำขอถัดไป)

ทำซ้ำคำขอเดิมทันที:

```bash
curl -I "http://<PUBLIC_IP>/"
```

Headers ที่คาดหวัง:

```
X-Cache-Status: HIT
X-CDN-Edge: cdn-simulator
```

`HIT` ยืนยันว่าการตอบสนองถูกให้บริการจาก NGINX disk cache โดยไม่ต้องติดต่อ origin

## การตรวจสอบไดเรกทอรีแคช

SSH เข้า VM เพื่อตรวจสอบว่ามีเนื้อหาที่แคชไว้อยู่บนดิสก์:

```bash
ssh azureuser@<PUBLIC_IP>

# ตรวจสอบว่าไดเรกทอรีแคชมีเนื้อหา
sudo find /var/cache/nginx/cdn -type f | head -20

# ตรวจสอบขนาดแคช
sudo du -sh /var/cache/nginx/cdn
```

## สถานะ NGINX

ตรวจสอบว่า NGINX กำลังทำงานและการกำหนดค่าถูกต้อง:

```bash
ssh azureuser@<PUBLIC_IP>

# ตรวจสอบสถานะบริการ
sudo systemctl status nginx

# ตรวจสอบความถูกต้องของการกำหนดค่า
sudo nginx -t

# ดูการเชื่อมต่อที่ใช้งานอยู่ (หากเปิดใช้ stub_status)
curl -s http://localhost/health
```

## การเชื่อมต่อ Origin

ตรวจสอบว่า edge node สามารถเข้าถึง origin server ได้:

```bash
ssh azureuser@<PUBLIC_IP>

# ทดสอบการเชื่อมต่อไปยัง origin (แทนที่ด้วย origin URL ของคุณ)
curl -I "https://your-origin.example.com/"
```

หากล้มเหลว ให้ตรวจสอบ:

1. ตัวแปร `origin_server` ถูกต้องใน Terraform config
2. Origin server อนุญาตการเชื่อมต่อขาเข้าจาก public IP ของ edge node
3. การแก้ไขชื่อ DNS ทำงานจาก VM (`nslookup your-origin.example.com`)

## ความคืบหน้า Cloud-Init

ติดตามความคืบหน้าการจัดเตรียมผ่าน cloud-init log:

```bash
ssh azureuser@<PUBLIC_IP> "tail -f /var/log/cloud-init-progress.log"
```

เฟสที่คาดหวัง: `[init]`, `[nic]` (การตรวจจับ NIC แบบไดนามิก), `[complete]`

หาก cloud-init รายงานข้อผิดพลาด:

```bash
ssh azureuser@<PUBLIC_IP> "sudo cloud-init status --long"
```

## การทดสอบแบบ End-to-End

รันวงจรคำขอแบบเต็มและตรวจสอบห่วงโซ่การตอบสนอง:

```bash
# คำขอแรก — MISS (ดึงจาก origin)
echo "=== Request 1 (expect MISS) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"

# คำขอที่สอง — HIT (ให้บริการจากแคช)
echo "=== Request 2 (expect HIT) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"
```

ผลลัพธ์ที่คาดหวัง:

```
=== Request 1 (expect MISS) ===
HTTP 200 | Cache: MISS
=== Request 2 (expect HIT) ===
HTTP 200 | Cache: HIT
```

## การตรวจสอบ Vendor Header

ตรวจสอบว่า CDN vendor headers ทั้งหมดถูกแทรกโดยการร้องขอ path `/headers` (เมื่อใช้ httpbin.org เป็น origin):

```bash
curl -s "http://<PUBLIC_IP>/headers" | python3 -m json.tool
```

การตอบสนองที่คาดหวังจะรวม headers จาก vendor ทั้งห้าราย:

```json
{
    "headers": {
        "True-Client-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Connecting-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Ipcountry": "US",
        "Cf-Ray": "<request_id>-SJC",
        "Cf-Bot-Score": "85",
        "Cf-Ja3-Hash": "e7d705a3286e19ea42f587b344ee6865",
        "Cloudfront-Viewer-Country": "US",
        "Cloudfront-Viewer-City": "San Jose",
        "Cloudfront-Is-Desktop-Viewer": "true",
        "Cloudfront-Is-Mobile-Viewer": "false",
        "Fastly-Client-Ip": "<YOUR_CLIENT_IP>",
        "X-Akamai-Edgescape": "georegion=263,country_code=US,...",
        "X-Azure-Clientip": "<YOUR_CLIENT_IP>",
        "X-Azure-Fdid": "a0a0a0a0-bbbb-cccc-dddd-e1e1e1e1e1e1",
        "X-Geo-Country-Code": "US"
    }
}
```

### การทดสอบการตรวจจับอุปกรณ์

ทดสอบด้วย User-Agent ของมือถือเพื่อตรวจสอบว่า headers การตรวจจับอุปกรณ์เปลี่ยนแปลง:

```bash
curl -s "http://<PUBLIC_IP>/headers?device=mobile" \
  -H "User-Agent: Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 like Mac OS X)" \
  | python3 -m json.tool | grep -E "Is-Mobile|Is-Desktop|is_mobile"
```

ผลลัพธ์ที่คาดหวัง:

```
"Cloudfront-Is-Desktop-Viewer": "false",
"Cloudfront-Is-Mobile-Viewer": "true",
```

## การแก้ไขปัญหา

### NGINX ไม่ทำงาน

```bash
ssh azureuser@<PUBLIC_IP>
sudo systemctl status nginx
sudo journalctl -u nginx --no-pager -n 50
```

### Cloud-init ยังไม่เสร็จ

หลังจาก `terraform apply` cloud-init ใช้เวลา 2-3 นาที ตรวจสอบความคืบหน้า:

```bash
ssh azureuser@<PUBLIC_IP>
sudo cloud-init status
sudo tail -f /var/log/cloud-init-output.log
```

### แคชแสดง MISS ตลอดเวลา

- ตรวจสอบว่า origin ส่งคืนรหัสตอบกลับที่แคชได้ (200, 301, 302)
- ตรวจสอบว่า origin ไม่ได้ส่ง headers `Cache-Control: no-cache` หรือ `no-store`
- ตรวจสอบ NGINX error logs: `sudo tail -f /var/log/nginx/error.log`

### การเชื่อมต่อถูกปฏิเสธบนพอร์ต 80/443

- ตรวจสอบว่ากฎ NSG ถูกนำไปใช้: `az network nsg rule list --nsg-name "$(terraform output -raw nsg_name)" --resource-group "$(terraform output -raw resource_group_name)" -o table`
- ตรวจสอบว่า NGINX กำลังรับฟังอยู่: `ssh azureuser@<PUBLIC_IP> "sudo ss -tlnp | grep nginx"`
