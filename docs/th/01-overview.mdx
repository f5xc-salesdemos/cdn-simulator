## วัตถุประสงค์

คอมโพเนนต์นี้จำลองโหนดขอบ CDN สำหรับสภาพแวดล้อมแล็บและสาธิต โดยแสดงบทบาทที่ผู้ให้บริการอย่าง Akamai, Cloudflare, Amazon CloudFront หรือ Fastly มีในสถาปัตยกรรมเครือข่ายของลูกค้า ซึ่งเป็นชั้นแคชที่อยู่ใกล้ผู้ใช้ปลายทางมากที่สุดและตั้งอยู่หน้าเซิร์ฟเวอร์ต้นทาง

ในสถาปัตยกรรมหลายผู้ให้บริการแบบ production ลูกค้ามักจะจับคู่ CDN ของบุคคลที่สามกับ F5 Distributed Cloud:

```
End User → CDN Edge (Akamai/Cloudflare/etc.) → F5 XC HTTP LB → Origin App
```

ตัวจำลองนี้แทนที่ CDN เชิงพาณิชย์ด้วยโหนดขอบที่ใช้ NGINX เพื่อให้สามารถสาธิตและทดสอบการผสานรวมได้โดยไม่ต้องมีสัญญาผู้ให้บริการหรือโครงสร้างพื้นฐาน production

## สถาปัตยกรรม

```
┌─────────┐     ┌──────────────────────┐     ┌─────────────────┐     ┌────────────┐
│  Client  │────▶│  CDN Edge (NGINX)    │────▶│  F5 XC HTTP LB  │────▶│ Origin App │
│          │     │  Ubuntu 24.04 Azure  │     │  (Origin Server) │     │            │
└─────────┘     │  - TLS termination   │     └─────────────────┘     └────────────┘
                │  - Disk-based cache  │
                │  - X-Cache-Status    │
                └──────────────────────┘
```

โหนดขอบ NGINX:

- **สิ้นสุดการเชื่อมต่อ TLS** ที่ขอบ (self-signed หรือ Let's Encrypt)
- **แคชการตอบกลับ** บนดิสก์โดยใช้ `proxy_cache_path`
- **ส่งต่อ cache miss** ไปยังเซิร์ฟเวอร์ต้นทางที่กำหนดค่าได้ (VIP ของ F5 XC HTTP load balancer)
- **รายงานสถานะแคช** ผ่าน response header `X-Cache-Status` (`HIT`, `MISS`, `BYPASS`, `EXPIRED`)

## สิ่งที่จำลอง

| ฟังก์ชัน CDN | การนำไปใช้งานด้วย NGINX |
|---|---|
| Edge caching | `proxy_cache` พร้อมการจัดเก็บบนดิสก์ |
| การสร้าง Cache key | `proxy_cache_key` อิงจาก scheme, host และ URI |
| Origin pull | `proxy_pass` ไปยัง F5 XC HTTP load balancer |
| การสิ้นสุด TLS | NGINX directive `ssl_certificate` |
| การเคารพ Cache-Control | `proxy_cache_valid` พร้อม origin header passthrough |
| การรายงานสถานะแคช | `add_header X-Cache-Status $upstream_cache_status` |
| Health endpoint | location `/health` ส่งกลับ 200 OK |

## Endpoints และพฤติกรรม Request/Response

### Health Check

```
GET /health
```

Response (200 OK, `Content-Type: application/json`):

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": ["akamai", "cloudflare", "cloudfront", "fastly", "azure-front-door"]
}
```

### CDN Proxy (เส้นทางอื่นทั้งหมด)

```
GET /<any-path>
```

Request headers ที่ถูกแทรกไปยังต้นทาง (67+ headers จาก 5 ผู้ให้บริการ):

| หมวดหมู่ | Headers ที่เพิ่ม |
|---|---|
| Client IP | `True-Client-IP`, `CF-Connecting-IP`, `Fastly-Client-IP`, `X-Azure-ClientIP`, `CloudFront-Viewer-Address`, `X-Forwarded-For`, `X-Real-IP` |
| ตำแหน่งทางภูมิศาสตร์ | `X-Akamai-Edgescape` (compound), `CF-IPCountry`, `cf-ipcity`, `cf-iplatitude/longitude`, `CloudFront-Viewer-Country/City/Latitude/Longitude`, `X-Geo-Country-Code/City/Region` |
| การตรวจจับอุปกรณ์ | `CloudFront-Is-Mobile-Viewer`, `CloudFront-Is-Desktop-Viewer`, `CloudFront-Is-Tablet-Viewer`, `X-Akamai-Device-Characteristics` |
| TLS/Fingerprint | `CloudFront-Viewer-TLS`, `cf-ja3-hash`, `cf-ja4`, `CloudFront-Viewer-JA3-Fingerprint` |
| การตรวจจับบอท | `cf-bot-score` (85 = น่าจะเป็นมนุษย์), `cf-verified-bot` |
| การติดตาม Request | `Cf-Ray`, `X-Akamai-Request-ID`, `X-Amz-Cf-Id`, `X-Azure-Ref` |
| Edge Identity | `X-CDN-Edge`, `X-CDN-POP`, `X-Served-By`, `Fastly-FF`, `X-Azure-FDID` |
| มาตรฐาน | `Via`, `Forwarded`, `CDN-Loop`, `X-Forwarded-Proto/Host/Port` |

Response headers ที่เพิ่มให้กับทุกการตอบกลับที่ผ่านพร็อกซี:

| Header | ค่า | วัตถุประสงค์ |
|---|---|---|
| `X-Cache-Status` | `HIT`, `MISS`, `BYPASS`, `EXPIRED`, `STALE` | พฤติกรรมแคชสำหรับ request นี้ |
| `X-CDN-Edge` | `cdn-simulator` | ระบุโหนดขอบนี้ |
| `X-CDN-POP` | `SJC` | รหัส IATA ของ Point of Presence ที่จำลอง |
| `X-Served-By` | `cache-sjc3120-SJC` | โหนดแคชจำลองในรูปแบบ Fastly |
| `X-Request-ID` | UUID (ต่อ request) | ตัวระบุ request ที่ไม่ซ้ำกัน |

### พฤติกรรมแคช

- Request แรกไปยังเส้นทางใดก็ตาม: `X-Cache-Status: MISS` (ดึงจากต้นทาง และถูกแคชแล้ว)
- Request ที่เหมือนกันครั้งถัดไป: `X-Cache-Status: HIT` (ให้บริการจากแคชบนดิสก์)
- Cache key: `$scheme$host$request_uri` (scheme + hostname + เส้นทางเต็ม + query string)
- Cache TTL: 10 นาทีสำหรับ 200/301/302, 1 นาทีสำหรับ 404
- การให้บริการแบบ stale: ส่งกลับเนื้อหาที่แคชไว้เมื่อต้นทางมีข้อผิดพลาด (500/502/503/504)

### การเข้าถึง VM

| วิธีการเข้าถึง | คำสั่ง/เส้นทาง |
|---|---|
| SSH | `ssh azureuser@<PUBLIC_IP>` |
| การกำหนดค่า NGINX | `/etc/nginx/conf.d/cdn-edge.conf` |
| NGINX logs | `/var/log/nginx/access.log` และ `/var/log/nginx/error.log` |
| ไดเรกทอรีแคช | `/var/cache/nginx/cdn/` |
| Cloud-init log | `/var/log/cloud-init-output.log` |

## การออกแบบคอมโพเนนต์แบบโมดูล

นี่เป็นส่วนหนึ่งของสภาพแวดล้อมแล็บที่ใหญ่กว่า แต่ละคอมโพเนนต์เป็นอิสระในตัวเองและถูกติดตั้งแยกกัน:

- **คอมโพเนนต์นี้** ให้บริการขอบ CDN (NGINX บน Azure VM)
- **คอมโพเนนต์อื่นๆ** ให้บริการแอปพลิเคชันต้นทาง, การกำหนดค่า F5 XC, DNS, นโยบาย WAF เป็นต้น

ผู้ปฏิบัติงานจะเพิ่มคอมโพเนนต์ทีละตัว เอกสารของแต่ละคอมโพเนนต์ถูกเขียนขึ้นเพื่อให้ผู้ช่วย AI สามารถอ่านและติดตั้งโครงสร้างพื้นฐานได้อย่างอัตโนมัติ

## ทำไมจึงเลือก NGINX

NGINX ถูกเลือกเป็นเอนจินจำลอง CDN เนื่องจาก:

1. **ผลิตภัณฑ์ของ F5** — F5 เข้าซื้อกิจการ NGINX Inc. ในปี 2019 ซึ่งเป็นส่วนหนึ่งของกลุ่มผลิตภัณฑ์ F5
2. **ได้รับการพิสูจน์ในอุตสาหกรรม** — Cloudflare ใช้ NGINX รัน CDN ทั้งหมดมากว่าทศวรรษก่อนจะย้ายไป Pingora; Netflix ใช้ NGINX สำหรับ CDN Open Connect ของพวกเขา
3. **กระบวนการเดียว** — จัดการการสิ้นสุด TLS และการแคชในไบนารีเดียว ไม่เหมือน Varnish ที่ต้องใช้ TLS proxy แยกต่างหาก
4. **ติดตั้งง่าย** — `apt install nginx` บน Ubuntu 24.04 เพียงสอง directive ก็เปิดใช้งานการแคชได้
5. **มีเอกสารครบถ้วน** — เอกสารอย่างเป็นทางการที่ครอบคลุมสำหรับการแคชเนื้อหาและการกำหนดค่า reverse proxy
