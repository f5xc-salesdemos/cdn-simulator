## Propósito

Este componente simula un nodo edge CDN para entornos de laboratorio y demostración. Representa el rol que proveedores como Akamai, Cloudflare, Amazon CloudFront o Fastly desempeñan en la arquitectura de red de un cliente — la capa de caché más cercana a los usuarios finales que se sitúa frente a un servidor de origen.

En arquitecturas de producción multi-proveedor, los clientes comúnmente combinan un CDN de terceros con F5 Distributed Cloud:

```
End User → CDN Edge (Akamai/Cloudflare/etc.) → F5 XC HTTP LB → Origin App
```

Este simulador reemplaza el CDN comercial con un nodo edge basado en NGINX para que la integración pueda ser demostrada y probada sin contratos de proveedores ni infraestructura de producción.

## Arquitectura

```
┌─────────┐     ┌──────────────────────┐     ┌─────────────────┐     ┌────────────┐
│  Client  │────▶│  CDN Edge (NGINX)    │────▶│  F5 XC HTTP LB  │────▶│ Origin App │
│          │     │  Ubuntu 24.04 Azure  │     │  (Origin Server) │     │            │
└─────────┘     │  - TLS termination   │     └─────────────────┘     └────────────┘
                │  - Disk-based cache  │
                │  - X-Cache-Status    │
                └──────────────────────┘
```

El nodo edge NGINX:

- **Termina TLS** en el edge (autofirmado o Let's Encrypt)
- **Almacena respuestas en caché** en disco usando `proxy_cache_path`
- **Reenvía fallos de caché** a un servidor de origen configurable (el VIP del balanceador de carga HTTP de F5 XC)
- **Reporta el estado de caché** mediante el encabezado de respuesta `X-Cache-Status` (`HIT`, `MISS`, `BYPASS`, `EXPIRED`)

## Qué simula este componente

| Función CDN | Implementación NGINX |
|---|---|
| Caché en el edge | `proxy_cache` con almacenamiento basado en disco |
| Generación de clave de caché | `proxy_cache_key` basada en esquema, host y URI |
| Obtención desde origen | `proxy_pass` al balanceador de carga HTTP de F5 XC |
| Terminación TLS | Directiva `ssl_certificate` de NGINX |
| Respeto de Cache-Control | `proxy_cache_valid` con paso de encabezados del origen |
| Reporte de estado de caché | `add_header X-Cache-Status $upstream_cache_status` |
| Endpoint de salud | Ubicación `/health` que retorna 200 OK |

## Endpoints y comportamiento de solicitud/respuesta

### Verificación de salud

```
GET /health
```

Respuesta (200 OK, `Content-Type: application/json`):

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": ["akamai", "cloudflare", "cloudfront", "fastly", "azure-front-door"]
}
```

### Proxy CDN (todas las demás rutas)

```
GET /<any-path>
```

Encabezados de solicitud inyectados hacia el origen (más de 67 encabezados de 5 proveedores):

| Categoría | Encabezados añadidos |
|---|---|
| IP del cliente | `True-Client-IP`, `CF-Connecting-IP`, `Fastly-Client-IP`, `X-Azure-ClientIP`, `CloudFront-Viewer-Address`, `X-Forwarded-For`, `X-Real-IP` |
| Geolocalización | `X-Akamai-Edgescape` (compuesto), `CF-IPCountry`, `cf-ipcity`, `cf-iplatitude/longitude`, `CloudFront-Viewer-Country/City/Latitude/Longitude`, `X-Geo-Country-Code/City/Region` |
| Detección de dispositivo | `CloudFront-Is-Mobile-Viewer`, `CloudFront-Is-Desktop-Viewer`, `CloudFront-Is-Tablet-Viewer`, `X-Akamai-Device-Characteristics` |
| TLS/Huella digital | `CloudFront-Viewer-TLS`, `cf-ja3-hash`, `cf-ja4`, `CloudFront-Viewer-JA3-Fingerprint` |
| Detección de bots | `cf-bot-score` (85 = probablemente humano), `cf-verified-bot` |
| Rastreo de solicitudes | `Cf-Ray`, `X-Akamai-Request-ID`, `X-Amz-Cf-Id`, `X-Azure-Ref` |
| Identidad del edge | `X-CDN-Edge`, `X-CDN-POP`, `X-Served-By`, `Fastly-FF`, `X-Azure-FDID` |
| Estándar | `Via`, `Forwarded`, `CDN-Loop`, `X-Forwarded-Proto/Host/Port` |

Encabezados de respuesta añadidos a cada respuesta proxied:

| Encabezado | Valores | Propósito |
|---|---|---|
| `X-Cache-Status` | `HIT`, `MISS`, `BYPASS`, `EXPIRED`, `STALE` | Comportamiento de caché para esta solicitud |
| `X-CDN-Edge` | `cdn-simulator` | Identifica este nodo edge |
| `X-CDN-POP` | `SJC` | Código IATA simulado del Punto de Presencia |
| `X-Served-By` | `cache-sjc3120-SJC` | Nodo de caché simulado en formato Fastly |
| `X-Request-ID` | UUID (por solicitud) | Identificador único de solicitud |

### Comportamiento de caché

- Primera solicitud a cualquier ruta: `X-Cache-Status: MISS` (obtenida del origen, ahora en caché)
- Solicitudes idénticas subsiguientes: `X-Cache-Status: HIT` (servida desde la caché en disco)
- Clave de caché: `$scheme$host$request_uri` (esquema + nombre de host + ruta completa + cadena de consulta)
- TTL de caché: 10 minutos para 200/301/302, 1 minuto para 404
- Servicio obsoleto: retorna contenido en caché ante errores del origen (500/502/503/504)

| Método de acceso | Comando/Ruta |

## Diseño modular de componentes

Este es una pieza de un entorno de laboratorio más amplio. Cada componente es autónomo y se despliega de forma independiente:

- **Este componente** proporciona el edge CDN (NGINX en VM de Azure)
- **Otros componentes** proporcionan la aplicación de origen, la configuración de F5 XC, DNS, políticas WAF, etc.

El operador humano añade componentes uno a la vez. La documentación de cada componente está escrita para que un asistente de IA pueda leerla y desplegar la infraestructura de forma autónoma.

## Por qué NGINX

NGINX fue seleccionado como el motor de simulación CDN porque:

1. **Producto de F5** — F5 adquirió NGINX Inc. en 2019; es parte del portafolio de F5
2. **Probado en la industria** — Cloudflare ejecutó todo su CDN sobre NGINX durante más de una década antes de migrar a Pingora; Netflix usa NGINX para su CDN Open Connect
3. **Proceso único** — gestiona la terminación TLS y el almacenamiento en caché en un solo binario, a diferencia de Varnish que requiere un proxy TLS separado
4. **Despliegue sencillo** — `apt install nginx` en Ubuntu 24.04, dos directivas habilitan el almacenamiento en caché
5. **Bien documentado** — documentación oficial extensa para almacenamiento en caché de contenido y configuración de proxy inverso
