Después del despliegue, ejecute estas comprobaciones para confirmar que el nodo edge del CDN está funcionando correctamente. Espere 2-3 minutos después de `terraform apply` para que cloud-init termine de instalar NGINX.

## Comprobación de salud

Verifique que NGINX está en ejecución y respondiendo:

```bash
curl -s "http://<PUBLIC_IP>/health" | jq .
```

Salida esperada:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

Si utiliza las salidas de Terraform:

```bash
curl -s "$(terraform output -raw health_check_url)" | jq .
```

## Cache Miss (Primera solicitud)

La primera solicitud a cualquier ruta será un cache miss — el edge obtiene el contenido del origen:

```bash
curl -I "http://<PUBLIC_IP>/"
```

Busque estas cabeceras en la respuesta:

```
X-Cache-Status: MISS
X-CDN-Edge: cdn-simulator
```

`MISS` significa que el contenido no estaba en caché y fue obtenido del servidor de origen.

## Cache Hit (Solicitud posterior)

Repita la misma solicitud inmediatamente:

```bash
curl -I "http://<PUBLIC_IP>/"
```

Cabeceras esperadas:

```
X-Cache-Status: HIT
X-CDN-Edge: cdn-simulator
```

`HIT` confirma que la respuesta fue servida desde la caché de disco de NGINX sin contactar al origen.

## Inspección del directorio de caché

Conéctese por SSH a la VM para verificar que el contenido en caché existe en disco:

```bash
ssh azureuser@<PUBLIC_IP>

# Verificar que el directorio de caché tiene contenido
sudo find /var/cache/nginx/cdn -type f | head -20

# Verificar el tamaño de la caché
sudo du -sh /var/cache/nginx/cdn
```

## Estado de NGINX

Verifique que NGINX está en ejecución y que la configuración es válida:

```bash
ssh azureuser@<PUBLIC_IP>

# Verificar el estado del servicio
sudo systemctl status nginx

# Validar la configuración
sudo nginx -t

# Ver conexiones activas (si stub_status está habilitado)
curl -s http://localhost/health
```

## Conectividad con el origen

Verifique que el nodo edge puede alcanzar el servidor de origen:

```bash
ssh azureuser@<PUBLIC_IP>

# Probar conectividad con el origen (reemplace con la URL de su origen)
curl -I "https://your-origin.example.com/"
```

Si esto falla, compruebe:

1. La variable `origin_server` es correcta en la configuración de Terraform
2. El servidor de origen permite conexiones entrantes desde la IP pública del nodo edge
3. La resolución DNS funciona desde la VM (`nslookup your-origin.example.com`)

## Progreso de Cloud-Init

Monitoree el progreso del aprovisionamiento a través del log de cloud-init:

```bash
ssh azureuser@<PUBLIC_IP> "tail -f /var/log/cloud-init-progress.log"
```

Fases esperadas: `[init]`, `[nic]` (detección dinámica de NIC), `[complete]`.

Si cloud-init reporta errores:

```bash
ssh azureuser@<PUBLIC_IP> "sudo cloud-init status --long"
```

## Prueba de extremo a extremo

Ejecute un ciclo completo de solicitudes y verifique la cadena de respuesta:

```bash
# Primera solicitud — MISS (obtiene del origen)
echo "=== Request 1 (expect MISS) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"

# Segunda solicitud — HIT (servida desde caché)
echo "=== Request 2 (expect HIT) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"
```

Salida esperada:

```
=== Request 1 (expect MISS) ===
HTTP 200 | Cache: MISS
=== Request 2 (expect HIT) ===
HTTP 200 | Cache: HIT
```

## Verificación de cabeceras de proveedor

Verifique que todas las cabeceras de proveedores CDN están siendo inyectadas solicitando la ruta `/headers` (cuando se usa httpbin.org como origen):

```bash
curl -s "http://<PUBLIC_IP>/headers" | python3 -m json.tool
```

La respuesta esperada incluye cabeceras de los cinco proveedores:

```json
{

        "True-Client-Ip": "<YOUR_CLIENT_IP>",
```

### Prueba de detección de dispositivo

Pruebe con un User-Agent móvil para verificar que las cabeceras de detección de dispositivo cambian:

```bash
curl -s "http://<PUBLIC_IP>/headers?device=mobile" \
  -H "User-Agent: Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 like Mac OS X)" \
  | python3 -m json.tool | grep -E "Is-Mobile|Is-Desktop|is_mobile"
```

Esperado:

```
"Cloudfront-Is-Desktop-Viewer": "false",
"Cloudfront-Is-Mobile-Viewer": "true",
```

## Solución de problemas

### NGINX no está en ejecución

```bash
ssh azureuser@<PUBLIC_IP>
sudo systemctl status nginx
sudo journalctl -u nginx --no-pager -n 50
```

### Cloud-init no ha terminado

Después de `terraform apply`, cloud-init tarda 2-3 minutos. Verifique el progreso:

```bash
ssh azureuser@<PUBLIC_IP>
sudo cloud-init status
sudo tail -f /var/log/cloud-init-output.log
```

### La caché siempre muestra MISS

- Verifique que el origen devuelve códigos de respuesta cacheables (200, 301, 302)
- Compruebe que el origen no envía cabeceras `Cache-Control: no-cache` o `no-store`
- Revise los logs de error de NGINX: `sudo tail -f /var/log/nginx/error.log`

### Conexión rechazada en el puerto 80/443

- Verifique que las reglas del NSG están aplicadas: `az network nsg rule list --nsg-name "$(terraform output -raw nsg_name)" --resource-group "$(terraform output -raw resource_group_name)" -o table`
- Verifique que NGINX está escuchando: `ssh azureuser@<PUBLIC_IP> "sudo ss -tlnp | grep nginx"`
