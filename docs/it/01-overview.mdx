## Scopo

Questo componente simula un nodo edge CDN per ambienti di laboratorio e dimostrativi. Rappresenta il ruolo che vendor come Akamai, Cloudflare, Amazon CloudFront o Fastly ricoprono nell'architettura di rete di un cliente вҖ” il livello di caching piГ№ vicino agli utenti finali, posizionato davanti a un server di origine.

Nelle architetture multi-vendor in produzione, i clienti comunemente abbinano un CDN di terze parti con F5 Distributed Cloud:

```
End User вҶ’ CDN Edge (Akamai/Cloudflare/etc.) вҶ’ F5 XC HTTP LB вҶ’ Origin App
```

Questo simulatore sostituisce il CDN commerciale con un nodo edge basato su NGINX, cosГ¬ che l'integrazione possa essere dimostrata e testata senza contratti con i vendor o infrastruttura di produzione.

Il nodo edge NGINX:

- **Termina il TLS** all'edge (certificato auto-firmato o Let's Encrypt)
- **Memorizza le risposte nella cache** su disco utilizzando `proxy_cache_path`
- **Inoltra i cache miss** a un server di origine configurabile (il VIP dell'HTTP load balancer F5 XC)
- **Riporta lo stato della cache** tramite l'header di risposta `X-Cache-Status` (`HIT`, `MISS`, `BYPASS`, `EXPIRED`)

## Cosa simula

| Funzione CDN | Implementazione NGINX |
|---|---|
| Caching all'edge | `proxy_cache` con storage su disco |
| Generazione della cache key | `proxy_cache_key` basata su scheme, host e URI |
| Origin pull | `proxy_pass` verso l'HTTP load balancer F5 XC |
| Terminazione TLS | Direttiva NGINX `ssl_certificate` |
| Rispetto del Cache-Control | `proxy_cache_valid` con passthrough degli header di origine |
| Reportistica dello stato della cache | `add_header X-Cache-Status $upstream_cache_status` |
| Endpoint di health | Location `/health` che restituisce 200 OK |

Header di risposta aggiunti a ogni risposta proxy:

| Header | Valori | Scopo |
|---|---|---|
| `X-Cache-Status` | `HIT`, `MISS`, `BYPASS`, `EXPIRED`, `STALE` | Comportamento della cache per questa richiesta |
| `X-CDN-Edge` | `cdn-simulator` | Identifica questo nodo edge |

| `X-Served-By` | `cache-sjc3120-SJC` | Nodo cache simulato in formato Fastly |
| `X-Request-ID` | UUID (per richiesta) | Identificatore univoco della richiesta |

### Comportamento della cache

- Prima richiesta a qualsiasi percorso: `X-Cache-Status: MISS` (recuperata dall'origine, ora in cache)
- Richieste identiche successive: `X-Cache-Status: HIT` (servita dalla cache su disco)
- Cache key: `$scheme$host$request_uri` (scheme + hostname + percorso completo + query string)
- TTL della cache: 10 minuti per 200/301/302, 1 minuto per 404
- Servizio stale: restituisce contenuto in cache in caso di errori dell'origine (500/502/503/504)

## PerchГ© NGINX

NGINX ГЁ stato selezionato come motore di simulazione CDN perchГ©:

1. **Prodotto F5** вҖ” F5 ha acquisito NGINX Inc. nel 2019; fa parte del portfolio F5
2. **Comprovato nel settore** вҖ” Cloudflare ha gestito l'intero CDN su NGINX per oltre un decennio prima di migrare a Pingora; Netflix utilizza NGINX per il proprio CDN Open Connect

5. **Ben documentato** вҖ” documentazione ufficiale estesa per il content caching e la configurazione del reverse proxy
