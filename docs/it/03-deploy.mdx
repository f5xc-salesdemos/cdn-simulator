Tutti i file Terraform si trovano nella directory [`terraform/`](https://github.com/f5xc-salesdemos/cdn-simulator/tree/main/terraform). Clonare il repository e distribuire direttamente:

```bash
git clone https://github.com/f5xc-salesdemos/cdn-simulator.git
cd cdn-simulator/terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and origin server
```

## Configurazione Terraform

### Struttura dei file Terraform

La directory terraform contiene 9 file che seguono lo standard Demo Resource Standard:

- `versions.tf` — Vincoli di versione per Terraform e il provider (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` — Configurazione dei provider Azure RM e Azure AD
- `data.tf` — Origini dati Azure AD per la risoluzione automatica del deployer
- `locals.tf` — Risoluzione del deployer, denominazione delle risorse Azure Cloud Adoption Framework, tag standard
- `main.tf` — Gruppo di risorse (denominato `rg-cdn-simulator-{environment}-{deployer}`)
- `variables.tf` — Tutte le variabili di input (3 obbligatorie, 8 facoltative)
- `network.tf` — VNet (10.100.0.0/16), subnet, IP pubblico, NSG (porte 22/80/443), NIC
- `vm.tf` — VM Ubuntu 24.04 con cloud-init tramite templatefile()
- `outputs.tf` — 17 output (15 standard + 2 specifici del componente)

`variables.tf` definisce 11 variabili di input organizzate nelle sezioni General, Compute e Component-Specific. L'identificatore `deployer` viene risolto automaticamente dall'account Azure AD — è necessario impostare solo `subscription_id`, `origin_server` e `origin_host`:

```hcl file=./_data/variables.tf
```

### Infrastruttura di rete

`network.tf` crea la VNet, la subnet, l'IP pubblico, l'NSG (porte 22/80/443) e la NIC:

```hcl file=./_data/network.tf
```

### Macchina virtuale con Cloud-Init

`vm.tf` crea la VM Ubuntu 24.04. Il percorso della chiave pubblica SSH viene espanso tramite `pathexpand()` per gestire `~`. Il template cloud-init riceve le variabili `origin_server` e `origin_host`:

```hcl file=./_data/vm.tf
```

### Provisioning con Cloud-Init

`cloud-init.yaml` fornisce alla VM l'ottimizzazione del kernel, i limiti systemd, NGINX con configurazione ottimizzata per le prestazioni, una zona chiavi cache da 128 MB, un pool keepalive upstream, la compressione gzip e oltre 67 intestazioni di vendor CDN. Una libreria helper condivisa (`/usr/local/lib/cloud-init-helpers.sh`) fornisce logica di ripetizione dei tentativi e registrazione dell'avanzamento in `/var/log/cloud-init-progress.log`.

Il cloud-init utilizza le variabili template di Terraform: `${origin_server}` e `${origin_host}` per la configurazione upstream. Le variabili NGINX come `${request_id}` vengono precedute da escape come `$${request_id}` nel templatefile di Terraform.

```yaml file=./_data/cloud-init.yaml
```

### Output

`outputs.tf` espone 17 output seguendo il Demo Resource Standard — 15 output standard condivisi da tutte le risorse demo (`deployer`, `public_ip`, `private_ip`, `ssh_command`, `resource_group_name`, `vm_name`, `nsg_name`, `vnet_name`, `subnet_id`, `component`, `environment`, `resource_group_id`, `vm_id`, `nsg_id`, `location`) più 2 output specifici del componente (`edge_url`, `health_check_url`):

```hcl file=./_data/outputs.tf
```

### File di variabili di esempio

Copiare `terraform.tfvars.example` in `terraform.tfvars` e inserire i propri valori. Il file `.gitignore` esclude `terraform.tfvars` per evitare il commit delle credenziali:

```hcl file=./_data/terraform.tfvars.example
```

## Distribuisci

```bash
# Initialize Terraform
terraform init

# Review the plan
terraform plan

# Apply
terraform apply
```

Terraform restituisce come output l'IP pubblico, il comando SSH e l'URL edge al termine della distribuzione.

## Dimensionamento della VM

Le VM della serie F ottimizzate per il calcolo sono consigliate per questo carico di lavoro NGINX proxy ad alto utilizzo di CPU. Il valore predefinito `Standard_F4s_v2` (4 vCPU, 8 GiB) è adatto per laboratori e utilizzo demo. Sovrascrivere la variabile `vm_size` per scenari di test di carico o benchmarking in produzione. L'ottimizzazione del kernel tramite cloud-init e la configurazione NGINX si adattano automaticamente al numero di vCPU (`worker_processes auto`).

## Post-distribuzione

Al termine di `terraform apply`, attendere 2-3 minuti affinché il cloud-init completi l'installazione e la configurazione di NGINX. Verificare l'endpoint di integrità:

```bash
curl -s "http://$(terraform output -raw public_ip)/health" | jq .
```

Risposta attesa:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

## Collegamento dai Componenti upstream

Il simulatore CDN richiede un server di origine già distribuito. Utilizzare gli output terraform del server di origine per popolare le variabili obbligatorie:

```bash
cd ../origin-server/terraform
origin_ip=$(terraform output -raw public_ip)

cd ../../cdn-simulator/terraform
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
origin_server   = "http://${origin_ip}"
origin_host     = "${origin_ip}:80"
EOF
```

| Variabile obbligatoria | Origine | Formato |
|---|---|---|
| `origin_server` | Output `public_ip` del Server di origine | `http://<ip>` (includere lo schema) |
| `origin_host` | Output `public_ip` del Server di origine | `<ip>:80` (senza schema, includere la porta) |

Procedere con [Configurazione NGINX](../04-nginx-config/) per le opzioni di personalizzazione oppure con [Verifica](../05-verify/) per il test della cache.
