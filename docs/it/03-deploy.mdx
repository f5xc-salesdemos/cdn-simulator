Tutti i file Terraform si trovano nella directory [`terraform/`](https://github.com/f5xc-salesdemos/cdn-simulator/tree/main/terraform). Clonate il repository ed eseguite il deploy direttamente:

```bash
git clone https://github.com/f5xc-salesdemos/cdn-simulator.git
cd cdn-simulator/terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and origin server
```

## Configurazione Terraform

### Struttura dei file Terraform

La directory terraform contiene 9 file che seguono il Demo Resource Standard:

- `versions.tf` — Vincoli di versione di Terraform e dei provider (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` — Configurazione dei provider Azure RM e Azure AD
- `data.tf` — Data source Azure AD per la risoluzione automatica del deployer
- `locals.tf` — Risoluzione del deployer, denominazione delle risorse secondo l'Azure Cloud Adoption Framework, tag standard
- `main.tf` — Resource group (denominato `rg-cdn-simulator-{environment}-{deployer}`)
- `variables.tf` — Tutte le variabili di input (3 obbligatorie, 8 opzionali)
- `network.tf` — VNet (10.100.0.0/16), subnet, IP pubblico, NSG (porte 22/80/443), NIC
- `vm.tf` — VM Ubuntu 24.04 con cloud-init tramite templatefile()
- `outputs.tf` — 17 output (15 standard + 2 specifici del componente)

`variables.tf` definisce 11 variabili di input organizzate nelle sezioni General, Compute e Component-Specific. L'identificatore `deployer` viene risolto automaticamente dal vostro account Azure AD — è necessario impostare solo `subscription_id`, `origin_server` e `origin_host`:

```hcl file=./_data/variables.tf
```

### Infrastruttura di rete

`network.tf` crea la VNet, la subnet, l'IP pubblico, il NSG (porte 22/80/443) e la NIC:

```hcl file=./_data/network.tf
```

### Macchina virtuale con Cloud-Init

`vm.tf` crea la VM Ubuntu 24.04. Il percorso della chiave pubblica SSH viene espanso tramite `pathexpand()` per gestire `~`. Il template cloud-init riceve le variabili `origin_server` e `origin_host`:

```hcl file=./_data/vm.tf
```

### Provisioning con Cloud-Init

`cloud-init.yaml` effettua il provisioning della VM con tuning del kernel, limiti systemd, NGINX con configurazione ottimizzata per le prestazioni, zona cache keys da 128 MB, pool upstream keepalive, compressione gzip e oltre 67 header di vendor CDN. Una libreria helper condivisa (`/usr/local/lib/cloud-init-helpers.sh`) fornisce logica di retry e logging dei progressi su `/var/log/cloud-init-progress.log`.

Il cloud-init utilizza variabili template di Terraform: `${origin_server}` e `${origin_host}` per la configurazione upstream. Le variabili NGINX come `${request_id}` vengono escaped come `$${request_id}` nel templatefile di Terraform.

```yaml file=./_data/cloud-init.yaml
```

### Output

`outputs.tf` espone 17 output seguendo il Demo Resource Standard — 15 output standard condivisi da tutte le risorse demo (`deployer`, `public_ip`, `private_ip`, `ssh_command`, `resource_group_name`, `vm_name`, `nsg_name`, `vnet_name`, `subnet_id`, `component`, `environment`, `resource_group_id`, `vm_id`, `nsg_id`, `location`) più 2 output specifici del componente (`edge_url`, `health_check_url`):

```hcl file=./_data/outputs.tf
```

### File di variabili di esempio

Copiate `terraform.tfvars.example` in `terraform.tfvars` e inserite i vostri valori. Il `.gitignore` esclude `terraform.tfvars` per evitare il commit delle credenziali:

```hcl file=./_data/terraform.tfvars.example
```

## Deploy

```bash
# Initialize Terraform
terraform init

# Review the plan
terraform plan

# Apply
terraform apply
```

Terraform restituisce l'IP pubblico, il comando SSH e l'URL dell'edge dopo un deploy riuscito.

## Dimensionamento della VM

Le VM compute-optimized della serie F sono consigliate per questo workload NGINX proxy ad alta intensità di CPU. La `Standard_F4s_v2` predefinita (4 vCPU, 8 GiB) è adatta per utilizzo in laboratorio e demo. Sovrascrivete la variabile `vm_size` per scenari di load testing o benchmarking in produzione. Il tuning del kernel e la configurazione NGINX del cloud-init scalano automaticamente con il numero di vCPU (`worker_processes auto`).

## Post-Deploy

Dopo il completamento di `terraform apply`, attendete 2-3 minuti affinché cloud-init termini l'installazione e la configurazione di NGINX. Verificate l'endpoint di health:

```bash
curl -s "http://$(terraform output -raw public_ip)/health" | jq .
```

Risposta attesa:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

## Collegamento dai componenti upstream

Il simulatore CDN richiede un origin server già distribuito. Utilizzate gli output terraform dell'origin server per popolare le variabili obbligatorie:

```bash
cd ../origin-server/terraform
origin_ip=$(terraform output -raw public_ip)

cd ../../cdn-simulator/terraform
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
origin_server   = "http://${origin_ip}"
origin_host     = "${origin_ip}:80"
EOF
```

| Variabile obbligatoria | Origine | Formato |
|---|---|---|
| `origin_server` | Output `public_ip` dell'origin server | `http://<ip>` (incluso lo schema) |
| `origin_host` | Output `public_ip` dell'origin server | `<ip>:80` (senza schema, inclusa la porta) |

Procedete alla [Configurazione NGINX](../04-nginx-config/) per le opzioni di personalizzazione o a [Verifica](../05-verify/) per il test della cache.
