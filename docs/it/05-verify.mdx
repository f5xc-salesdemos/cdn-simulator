Dopo il deployment, eseguire questi controlli per confermare che il nodo edge CDN funzioni correttamente. Attendere 2-3 minuti dopo `terraform apply` affinché cloud-init completi l'installazione di NGINX.

## Controllo dello stato di salute

Verificare che NGINX sia in esecuzione e risponda:

```bash
curl -s "http://<PUBLIC_IP>/health" | jq .
```

Output atteso:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

Se si utilizzano gli output di Terraform:

```bash
curl -s "$(terraform output -raw health_check_url)" | jq .
```

## Cache Miss (Prima richiesta)

La prima richiesta a qualsiasi percorso sarà un cache miss — il nodo edge recupera il contenuto dall'origin:

```bash
curl -I "http://<PUBLIC_IP>/"
```

Cercare questi header nella risposta:

```
X-Cache-Status: MISS
X-CDN-Edge: cdn-simulator
```

`MISS` significa che il contenuto non era presente nella cache ed è stato recuperato dal server di origine.

## Cache Hit (Richiesta successiva)

Ripetere immediatamente la stessa richiesta:

```bash
curl -I "http://<PUBLIC_IP>/"
```

Header attesi:

```
X-Cache-Status: HIT
X-CDN-Edge: cdn-simulator
```

`HIT` conferma che la risposta è stata servita dalla cache su disco di NGINX senza contattare l'origin.

## Ispezione della directory di cache

Accedere via SSH alla VM per verificare che il contenuto memorizzato nella cache esista su disco:

```bash
ssh azureuser@<PUBLIC_IP>

# Verificare che la directory di cache contenga dei dati
sudo find /var/cache/nginx/cdn -type f | head -20

# Verificare la dimensione della cache
sudo du -sh /var/cache/nginx/cdn
```

## Stato di NGINX

Verificare che NGINX sia in esecuzione e che la configurazione sia valida:

```bash
ssh azureuser@<PUBLIC_IP>

# Verificare lo stato del servizio
sudo systemctl status nginx

# Validare la configurazione
sudo nginx -t

# Visualizzare le connessioni attive (se stub_status è abilitato)
curl -s http://localhost/health
```

## Connettività verso l'origin

Verificare che il nodo edge possa raggiungere il server di origine:

```bash
ssh azureuser@<PUBLIC_IP>

# Testare la connettività verso l'origin (sostituire con il proprio URL di origine)
curl -I "https://your-origin.example.com/"
```

In caso di errore, verificare:

1. Che la variabile `origin_server` sia corretta nella configurazione Terraform
2. Che il server di origine consenta connessioni in ingresso dall'IP pubblico del nodo edge
3. Che la risoluzione DNS funzioni dalla VM (`nslookup your-origin.example.com`)

## Avanzamento di Cloud-Init

Monitorare l'avanzamento del provisioning tramite il log di cloud-init:

```bash
ssh azureuser@<PUBLIC_IP> "tail -f /var/log/cloud-init-progress.log"
```

Fasi attese: `[init]`, `[nic]` (rilevamento dinamico della NIC), `[complete]`.

Se cloud-init segnala errori:

```bash
ssh azureuser@<PUBLIC_IP> "sudo cloud-init status --long"
```

## Test end-to-end

Eseguire un ciclo completo di richieste e verificare la catena di risposta:

```bash
# Prima richiesta — MISS (recupera dall'origin)
echo "=== Request 1 (expect MISS) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"

# Seconda richiesta — HIT (servita dalla cache)
echo "=== Request 2 (expect HIT) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"
```

Output atteso:

```
=== Request 1 (expect MISS) ===
HTTP 200 | Cache: MISS
=== Request 2 (expect HIT) ===
HTTP 200 | Cache: HIT
```

## Verifica degli header dei vendor

Verificare che tutti gli header dei vendor CDN vengano iniettati richiedendo il percorso `/headers` (quando si utilizza httpbin.org come origin):

```bash
curl -s "http://<PUBLIC_IP>/headers" | python3 -m json.tool
```

La risposta attesa include gli header di tutti e cinque i vendor:

```json
{
    "headers": {
        "True-Client-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Connecting-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Ipcountry": "US",
        "Cf-Ray": "<request_id>-SJC",
        "Cf-Bot-Score": "85",
        "Cf-Ja3-Hash": "e7d705a3286e19ea42f587b344ee6865",
        "Cloudfront-Viewer-Country": "US",
        "Cloudfront-Viewer-City": "San Jose",
        "Cloudfront-Is-Desktop-Viewer": "true",
```

### Test di rilevamento del dispositivo

Testare con uno User-Agent mobile per verificare che gli header di rilevamento del dispositivo cambino:

```bash
curl -s "http://<PUBLIC_IP>/headers?device=mobile" \
  -H "User-Agent: Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 like Mac OS X)" \
  | python3 -m json.tool | grep -E "Is-Mobile|Is-Desktop|is_mobile"
```

Risultato atteso:

```
"Cloudfront-Is-Desktop-Viewer": "false",
"Cloudfront-Is-Mobile-Viewer": "true",
```

## Risoluzione dei problemi

### NGINX non è in esecuzione

```bash
ssh azureuser@<PUBLIC_IP>
sudo systemctl status nginx
sudo journalctl -u nginx --no-pager -n 50
```

### Cloud-init non è terminato

Dopo `terraform apply`, cloud-init richiede 2-3 minuti. Verificare l'avanzamento:

```bash
ssh azureuser@<PUBLIC_IP>
sudo cloud-init status
sudo tail -f /var/log/cloud-init-output.log
```

### La cache mostra sempre MISS

- Verificare che l'origin restituisca codici di risposta memorizzabili nella cache (200, 301, 302)
- Verificare che l'origin non invii header `Cache-Control: no-cache` o `no-store`
- Controllare i log degli errori di NGINX: `sudo tail -f /var/log/nginx/error.log`

### Connessione rifiutata sulla porta 80/443

- Verificare che le regole NSG siano applicate: `az network nsg rule list --nsg-name "$(terraform output -raw nsg_name)" --resource-group "$(terraform output -raw resource_group_name)" -o table`
- Verificare che NGINX sia in ascolto: `ssh azureuser@<PUBLIC_IP> "sudo ss -tlnp | grep nginx"`
