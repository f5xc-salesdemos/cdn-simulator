Dopo il deployment, esegui questi controlli per confermare che il nodo edge CDN funzioni correttamente. Attendi 2-3 minuti dopo `terraform apply` affinché cloud-init completi l'installazione di NGINX.

## Controllo di Integrità

Verifica che NGINX sia in esecuzione e risponda:

```bash
curl -s "http://<PUBLIC_IP>/health" | jq .
```

Output atteso:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

Se utilizzi gli output di Terraform:

```bash
curl -s "$(terraform output -raw health_check_url)" | jq .
```

## Cache Miss (Prima Richiesta)

La prima richiesta a qualsiasi percorso sarà un cache miss — il nodo edge recupera il contenuto dall'origin:

```bash
curl -I "http://<PUBLIC_IP>/"
```

Cerca questi header nella risposta:

```
X-Cache-Status: MISS
X-CDN-Edge: cdn-simulator
```

`MISS` significa che il contenuto non era in cache ed è stato recuperato dal server origin.

## Cache Hit (Richiesta Successiva)

Ripeti la stessa richiesta immediatamente:

```bash
curl -I "http://<PUBLIC_IP>/"
```

Header attesi:

```
X-Cache-Status: HIT
X-CDN-Edge: cdn-simulator
```

`HIT` conferma che la risposta è stata servita dalla cache su disco di NGINX senza contattare l'origin.

## Ispezione della Directory Cache

Connettiti via SSH alla VM per verificare che il contenuto memorizzato in cache esista su disco:

```bash
ssh azureuser@<PUBLIC_IP>

# Controlla che la directory cache contenga contenuti
sudo find /var/cache/nginx/cdn -type f | head -20

# Controlla la dimensione della cache
sudo du -sh /var/cache/nginx/cdn
```

## Stato di NGINX

Verifica che NGINX sia in esecuzione e che la configurazione sia valida:

```bash
ssh azureuser@<PUBLIC_IP>

# Controlla lo stato del servizio
sudo systemctl status nginx

# Valida la configurazione
sudo nginx -t

# Visualizza le connessioni attive (se stub_status è abilitato)
curl -s http://localhost/health
```

## Connettività all'Origin

Verifica che il nodo edge possa raggiungere il server origin:

```bash
ssh azureuser@<PUBLIC_IP>

# Testa la connettività all'origin (sostituisci con il tuo URL origin)
curl -I "https://your-origin.example.com/"
```

Se questo fallisce, controlla:

1. La variabile `origin_server` è corretta nella configurazione Terraform
2. Il server origin consente connessioni in entrata dall'IP pubblico del nodo edge
3. La risoluzione DNS funziona dalla VM (`nslookup your-origin.example.com`)

## Avanzamento di Cloud-Init

Monitora l'avanzamento del provisioning tramite il log di cloud-init:

```bash
ssh azureuser@<PUBLIC_IP> "tail -f /var/log/cloud-init-progress.log"
```

Fasi attese: `[init]`, `[nic]` (rilevamento dinamico della NIC), `[complete]`.

Se cloud-init riporta errori:

```bash
ssh azureuser@<PUBLIC_IP> "sudo cloud-init status --long"
```

## Test End-to-End

Esegui un ciclo completo di richieste e verifica la catena di risposta:

```bash
# Prima richiesta — MISS (recupera dall'origin)
echo "=== Request 1 (expect MISS) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"

# Seconda richiesta — HIT (servita dalla cache)
echo "=== Request 2 (expect HIT) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"
```

Output atteso:

```
=== Request 1 (expect MISS) ===
HTTP 200 | Cache: MISS
=== Request 2 (expect HIT) ===
HTTP 200 | Cache: HIT
```

## Verifica degli Header Vendor

Verifica che tutti gli header dei vendor CDN vengano iniettati richiedendo il percorso `/headers` (quando si utilizza httpbin.org come origin):

```bash
curl -s "http://<PUBLIC_IP>/headers" | python3 -m json.tool
```

La risposta attesa include header da tutti e cinque i vendor:

```json
{
    "headers": {
        "True-Client-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Connecting-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Ipcountry": "US",
        "Cf-Ray": "<request_id>-SJC",
        "Cf-Bot-Score": "85",
        "Cf-Ja3-Hash": "e7d705a3286e19ea42f587b344ee6865",
        "Cloudfront-Viewer-Country": "US",
        "Cloudfront-Viewer-City": "San Jose",
        "Cloudfront-Is-Desktop-Viewer": "true",
```

### Test di Rilevamento Dispositivo

Testa con uno User-Agent mobile per verificare che gli header di rilevamento dispositivo cambino:

```bash
curl -s "http://<PUBLIC_IP>/headers?device=mobile" \
  -H "User-Agent: Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 like Mac OS X)" \
  | python3 -m json.tool | grep -E "Is-Mobile|Is-Desktop|is_mobile"
```

Output atteso:

```
"Cloudfront-Is-Desktop-Viewer": "false",
"Cloudfront-Is-Mobile-Viewer": "true",
```

## Risoluzione dei Problemi

### NGINX non in esecuzione

```bash
ssh azureuser@<PUBLIC_IP>
sudo systemctl status nginx
sudo journalctl -u nginx --no-pager -n 50
```

### Cloud-init non completato

Dopo `terraform apply`, cloud-init impiega 2-3 minuti. Controlla l'avanzamento:

```bash
ssh azureuser@<PUBLIC_IP>
sudo cloud-init status
sudo tail -f /var/log/cloud-init-output.log
```

### La cache mostra sempre MISS

- Verifica che l'origin restituisca codici di risposta memorizzabili in cache (200, 301, 302)
- Controlla che l'origin non invii header `Cache-Control: no-cache` o `no-store`
- Controlla i log degli errori di NGINX: `sudo tail -f /var/log/nginx/error.log`

### Connessione rifiutata sulla porta 80/443

- Verifica che le regole NSG siano applicate: `az network nsg rule list --nsg-name "$(terraform output -raw nsg_name)" --resource-group "$(terraform output -raw resource_group_name)" -o table`
- Verifica che NGINX sia in ascolto: `ssh azureuser@<PUBLIC_IP> "sudo ss -tlnp | grep nginx"`
