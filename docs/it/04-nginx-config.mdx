Il provisioning cloud-init distribuisce una configurazione edge CDN NGINX completamente ottimizzata per le prestazioni. Questa pagina documenta ogni livello di configurazione, dall'ottimizzazione del kernel al comportamento della cache fino all'iniezione degli header dei vendor. Tutte le impostazioni sono state verificate con un test di carico continuo di 48 ore con picchi di 172.540 req/s.

## Configurazione principale NGINX

### Worker e connessioni

```nginx
worker_processes auto;           # 4 workers on D4s_v5 (1 per vCPU)
worker_rlimit_nofile 65535;      # Per-worker file descriptor limit

events {
    use epoll;                   # Linux-optimized event model
    worker_connections 8192;     # 4 workers x 8192 = 32,768 max concurrent
    multi_accept on;             # Accept all pending connections per event loop
    accept_mutex off;            # Not needed with epoll + reuseport
}
```

L'override di systemd in `/etc/systemd/system/nginx.service.d/override.conf` imposta `LimitNOFILE=65535` per la corrispondenza.

### Buffer proxy

```nginx
proxy_buffering on;
proxy_buffer_size 16k;           # Header buffer (handles large CDN headers)
proxy_buffers 64 16k;            # 1 MB per connection (64 x 16k)
proxy_busy_buffers_size 256k;    # Can send 256k to client while still reading
```

La dimensione del busy buffer di 256k Ă¨ critica â€” deve superare la risposta in cache piĂą grande (Juice Shop Ă¨ 75 KB). L'impostazione originale di 64k causava serializzazione sotto carico.

### Compressione Gzip

```nginx
gzip on;
gzip_comp_level 4;               # Balance: ~85% of max compression at ~40% CPU
gzip_min_length 256;             # Skip tiny responses (gzip header overhead)
gzip_vary on;                    # Vary: Accept-Encoding for correct caching
gzip_proxied any;                # Compress all proxied responses
gzip_types text/plain text/css text/javascript text/xml
           application/json application/javascript application/xml
           application/xml+rss application/atom+xml
           application/ld+json application/manifest+json
           image/svg+xml;
```

### Cache dei file aperti

```nginx
open_file_cache max=200000 inactive=20s;
open_file_cache_valid 30s;
open_file_cache_min_uses 2;
open_file_cache_errors on;
```

Memorizza in cache i file descriptor e i metadati degli oggetti nella cache, eliminando le syscall `stat()` e `open()` sui file piĂą richiesti.

### Keepalive del client

```nginx
keepalive_timeout 65;
keepalive_requests 100000;       # Was 1000 â€” caused connection recycling at 90K req/s
```

A 90K req/s con `keepalive_requests 1000`, le connessioni si riciclavano ogni 11 secondi, generando socket TIME_WAIT. L'aumento a 100.000 ha eliminato completamente questo problema.

### Logging degli accessi

```nginx
log_format cdn '$remote_addr [$time_local] "$request" $status $body_bytes_sent $upstream_cache_status $request_time';
access_log /var/log/nginx/access.log cdn buffer=256k flush=5s;
```

Il formato di log `cdn` include `$upstream_cache_status` (HIT/MISS) e `$request_time` per l'analisi della latenza. Il logging con buffer (256k/flush ogni 5s) riduce l'overhead di I/O sotto carico elevato.

## Keepalive upstream

```nginx
upstream origin_backend {
    server 20.12.78.159:80;
    keepalive 256;               # Persistent connections per worker to origin
    keepalive_timeout 60s;
    keepalive_requests 1000;
}
```

Con 4 worker x 256 keepalive = 1.024 connessioni calde verso l'origin. Combinato con `proxy_http_version 1.1` e `proxy_set_header Connection ""`, questo elimina l'overhead dell'handshake TCP ad ogni cache miss.

## Configurazione della cache

### Percorso della cache

```nginx
proxy_cache_path /var/cache/nginx/cdn
                 levels=1:2
                 keys_zone=cdn_cache:32m
                 max_size=25g
                 inactive=24h
                 use_temp_path=off;
```

| Parametro | Valore | Scopo |
|---|---|---|
| `keys_zone=cdn_cache:32m` | 32 MB di memoria condivisa | Memorizza circa 256.000 chiavi di cache e metadati |
| `max_size=25g` | Limite disco di 25 GB | Utilizza la maggior parte del disco OS da 30 GB; evizione LRU al raggiungimento del limite |
| `inactive=24h` | Timeout di inattivitĂ  di 24 ore | Il contenuto non accesso per 24 ore viene rimosso (non Ă¨ la stessa cosa del TTL) |
| `use_temp_path=off` | Scrittura diretta nella directory della cache | Evita spostamenti di file tra filesystem diversi |

### Comportamento della cache

```nginx
proxy_cache cdn_cache;
proxy_cache_valid 200 301 302 4h;
proxy_cache_valid 404 1m;
proxy_cache_key "$scheme$host$request_uri";
proxy_cache_lock on;
proxy_cache_lock_age 3s;
proxy_cache_lock_timeout 3s;
proxy_cache_background_update on;
proxy_cache_use_stale updating error timeout http_500 http_502 http_503 http_504;
proxy_ignore_headers Set-Cookie Cache-Control Expires Vary;

proxy_hide_header X-Cache-Status;
proxy_hide_header Vary;
```

| Direttiva | Valore | Scopo |
|---|---|---|
| `proxy_cache_valid 200 301 302 4h` | TTL di 4 ore | Il contenuto in cache Ă¨ valido per 4 ore (aumentato per ridurre i cali di throughput durante le ondate di aggiornamento) |
| `proxy_cache_lock on` | Prevenzione del thundering herd | Solo 1 richiesta va all'origin per ogni URL non in cache; le altre attendono il riempimento della cache |
| `proxy_cache_lock_age 3s` | Timeout del lock | Se la prima richiesta non Ă¨ completata in 3s, ne viene consentita un'altra |
| `proxy_cache_background_update on` | Aggiornamento a latenza zero | Serve immediatamente il contenuto stale mentre aggiorna in background |
| `proxy_cache_use_stale` | Resilienza | Serve contenuto stale durante errori dell'origin (500/502/503/504) o durante l'aggiornamento |
| `proxy_ignore_headers` | Forzare il caching | Ignora Set-Cookie, Cache-Control, Expires, Vary dall'origin â€” il CDN decide il TTL e il comportamento di Vary (Juice Shop invia `max-age=0` e tripli header `Vary` che impedivano un caching efficace) |
| `proxy_hide_header X-Cache-Status` | Rimozione header dell'origin | L'NGINX di origin aggiunge il proprio X-Cache-Status â€” viene rimosso affinchĂ© sia visibile solo lo stato della cache del CDN |
| `proxy_hide_header Vary` | Prevenzione della frammentazione della cache | L'origin invia molteplici header `Vary: Accept-Encoding`. La loro rimozione previene la frammentazione delle chiavi di cache tra le permutazioni di `Accept-Encoding`. La direttiva `gzip_vary on` di NGINX aggiunge automaticamente il singolo header `Vary` corretto |

### Timeout del proxy

```nginx
proxy_read_timeout 30s;
proxy_connect_timeout 10s;
proxy_send_timeout 15s;
```

Concede all'origin piĂą tempo di risposta sotto carico, fallendo comunque rapidamente in caso di problemi di connessione.

### Blocco server

```nginx
server {
    listen 80 reuseport;        # Kernel distributes connections across all 4 workers
    server_name _;
}
```

`reuseport` abilita `SO_REUSEPORT` â€” il kernel distribuisce le connessioni in ingresso direttamente ai processi worker, eliminando la contesa dell'accept mutex.

## Rilevamento dispositivi

Il simulatore rileva il tipo di dispositivo dall'header `User-Agent` utilizzando le direttive `map` di NGINX:

- **Mobile**: Corrisponde a iPhone, Android (non tablet), iPod, BlackBerry, Opera Mini, IEMobile
- **Tablet**: Corrisponde a iPad, tablet Android, Kindle, PlayBook
- **Desktop**: Default quando non corrisponde nĂ© mobile nĂ© tablet

Il tipo di dispositivo si riflette in:
- `CloudFront-Is-Desktop-Viewer` / `CloudFront-Is-Mobile-Viewer` / `CloudFront-Is-Tablet-Viewer`
- `X-Akamai-Device-Characteristics` (campi `is_mobile`, `is_tablet`, `is_wireless_device`)

## Operazioni

### Modifica del server di origine

```bash
ssh azureuser@<PUBLIC_IP>

# Update upstream server
sudo sed -i 's|server .*;|server NEW_HOST:80;|' /etc/nginx/conf.d/cdn-edge.conf

# Clear cache and reload
sudo rm -rf /var/cache/nginx/cdn/*
sudo nginx -t && sudo systemctl reload nginx
```

In alternativa, aggiornare `origin_host` in `terraform.tfvars` ed eseguire `terraform apply` per il reprovisioning.

### Pulizia della cache

```bash
ssh azureuser@<PUBLIC_IP>
sudo rm -rf /var/cache/nginx/cdn/*
sudo systemctl reload nginx
```

### Verifica delle statistiche della cache

```bash
# Object count and disk usage
sudo find /var/cache/nginx/cdn -type f | wc -l
sudo du -sh /var/cache/nginx/cdn

# Recent access log with cache status
tail -20 /var/log/nginx/access.log
```

### Monitoraggio sotto carico

```bash
# Real-time connections and socket states
ss -s

# NGINX worker CPU usage
top -bn1 | grep nginx

# Upstream keepalive connections
ss -tn state established dst <ORIGIN_IP> | wc -l

# TIME_WAIT socket count
ss -tn state time-wait | wc -l
```
