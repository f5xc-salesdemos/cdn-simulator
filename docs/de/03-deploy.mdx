Alle Terraform-Dateien befinden sich im Verzeichnis [`terraform/`](https://github.com/f5xc-salesdemos/cdn-simulator/tree/main/terraform). Repository klonen und direkt bereitstellen:

```bash
git clone https://github.com/f5xc-salesdemos/cdn-simulator.git
cd cdn-simulator/terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and origin server
```

## Terraform-Konfiguration

### Terraform-Dateistruktur

Das Terraform-Verzeichnis enthält 9 Dateien gemäß dem Demo-Ressourcen-Standard:

- `versions.tf` — Terraform- und Provider-Versionseinschränkungen (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` — Azure RM- und Azure AD-Provider-Konfiguration
- `data.tf` — Azure AD-Datenquellen für die automatische Auflösung des Deployers
- `locals.tf` — Deployer-Auflösung, Azure Cloud Adoption Framework-Ressourcenbenennung, Standard-Tags
- `main.tf` — Ressourcengruppe (benannt `rg-cdn-simulator-{environment}-{deployer}`)
- `variables.tf` — Alle Eingabevariablen (3 erforderlich, 8 optional)
- `network.tf` — VNet (10.100.0.0/16), Subnetz, öffentliche IP, NSG (Ports 22/80/443), NIC
- `vm.tf` — Ubuntu 24.04 VM mit cloud-init über templatefile()
- `outputs.tf` — 17 Ausgaben (15 Standard + 2 komponentenspezifisch)

`variables.tf` definiert 11 Eingabevariablen, gegliedert in die Abschnitte Allgemein, Compute und Komponentenspezifisch. Die `deployer`-Kennung wird automatisch aus Ihrem Azure AD-Konto aufgelöst – Sie müssen lediglich `subscription_id`, `origin_server` und `origin_host` festlegen:

```hcl file=../_data/variables.tf
```

### Netzwerkinfrastruktur

`network.tf` erstellt das VNet, Subnetz, die öffentliche IP, NSG (Ports 22/80/443) und NIC:

```hcl file=../_data/network.tf
```

### Virtuelle Maschine mit Cloud-Init

`vm.tf` erstellt die Ubuntu 24.04 VM. Der SSH-Public-Key-Pfad wird über `pathexpand()` erweitert, um `~` zu verarbeiten. Das Cloud-init-Template erhält die Variablen `origin_server` und `origin_host`:

```hcl file=../_data/vm.tf
```

### Cloud-Init-Bereitstellung

`cloud-init.yaml` stellt die VM mit Kernel-Tuning, systemd-Limits, NGINX mit leistungsoptimierter Konfiguration, 128 MB Cache-Keys-Zone, Upstream-Keepalive-Pool, gzip-Komprimierung und 67+ CDN-Vendor-Headern bereit. Eine gemeinsame Hilfsbibliothek (`/usr/local/lib/cloud-init-helpers.sh`) stellt Wiederholungslogik und Fortschrittsprotokollierung in `/var/log/cloud-init-progress.log` bereit.

Das Cloud-init verwendet Terraform-Template-Variablen: `${origin_server}` und `${origin_host}` für die Upstream-Konfiguration. NGINX-Variablen wie `${request_id}` werden in der Terraform-templatefile als `$${request_id}` escaped.

```yaml file=../_data/cloud-init.yaml
```

### Ausgaben

`outputs.tf` stellt 17 Ausgaben gemäß dem Demo-Ressourcen-Standard bereit – 15 Standard-Ausgaben, die von allen Demo-Ressourcen gemeinsam genutzt werden (`deployer`, `public_ip`, `private_ip`, `ssh_command`, `resource_group_name`, `vm_name`, `nsg_name`, `vnet_name`, `subnet_id`, `component`, `environment`, `resource_group_id`, `vm_id`, `nsg_id`, `location`) sowie 2 komponentenspezifische Ausgaben (`edge_url`, `health_check_url`):

```hcl file=../_data/outputs.tf
```

### Beispiel-Variablendatei

Kopieren Sie `terraform.tfvars.example` nach `terraform.tfvars` und tragen Sie Ihre Werte ein. Die `.gitignore` schließt `terraform.tfvars` aus, um das versehentliche Committen von Zugangsdaten zu verhindern:

```hcl file=../_data/terraform.tfvars.example
```

## Bereitstellen

```bash
# Initialize Terraform
terraform init

# Review the plan
terraform plan

# Apply
terraform apply
```

Terraform gibt nach erfolgreicher Bereitstellung die öffentliche IP, den SSH-Befehl und die Edge-URL aus.

## VM-Dimensionierung

Die compute-optimierten VMs der F-Serie werden für diese CPU-gebundene NGINX-Proxy-Arbeitslast empfohlen. Die Standardgröße `Standard_F4s_v2` (4 vCPU, 8 GiB) ist für Labor- und Demo-Zwecke geeignet. Überschreiben Sie die Variable `vm_size` für Lasttest- oder Produktions-Benchmarking-Szenarien. Das Kernel-Tuning per cloud-init und die NGINX-Konfiguration skalieren automatisch mit der vCPU-Anzahl (`worker_processes auto`).

## Nach der Bereitstellung

Nachdem `terraform apply` abgeschlossen ist, warten Sie 2–3 Minuten, bis cloud-init die Installation und Konfiguration von NGINX abgeschlossen hat. Überprüfen Sie den Health-Endpunkt:

```bash
curl -s "http://$(terraform output -raw public_ip)/health" | jq .
```

Erwartete Antwort:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

## Verbindung mit vorgelagerten Komponenten

Der CDN-Simulator benötigt einen bereitgestellten Ursprungsserver. Verwenden Sie die Terraform-Ausgaben des Ursprungsservers, um die erforderlichen Variablen zu befüllen:

```bash
cd ../origin-server/terraform
origin_ip=$(terraform output -raw public_ip)

cd ../../cdn-simulator/terraform
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
origin_server   = "http://${origin_ip}"
origin_host     = "${origin_ip}:80"
EOF
```

| Erforderliche Variable | Quelle | Format |
|---|---|---|
| `origin_server` | Ursprungsserver-Ausgabe `public_ip` | `http://<ip>` (Schema angeben) |
| `origin_host` | Ursprungsserver-Ausgabe `public_ip` | `<ip>:80` (kein Schema, Port angeben) |

Fahren Sie mit [NGINX-Konfiguration](../04-nginx-config/) für Anpassungsoptionen oder [Verifizieren](../05-verify/) für Cache-Tests fort.
