Der NGINX-Edge-Node:

- **Terminiert TLS** am Edge (selbstsigniert oder Let's Encrypt)
- **Cached Antworten** auf der Festplatte mittels `proxy_cache_path`
- **Leitet Cache-Misses weiter** an einen konfigurierbaren Origin-Server (die F5 XC HTTP Load Balancer VIP)
- **Meldet den Cache-Status** Гјber den `X-Cache-Status`-Response-Header (`HIT`, `MISS`, `BYPASS`, `EXPIRED`)

## Was simuliert wird

| CDN-Funktion | NGINX-Implementierung |
|---|---|
| Edge-Caching | `proxy_cache` mit festplattenbasiertem Speicher |
| Cache-Key-Generierung | `proxy_cache_key` basierend auf Schema, Host und URI |
| Origin-Pull | `proxy_pass` zum F5 XC HTTP Load Balancer |
| TLS-Terminierung | NGINX `ssl_certificate`-Direktive |
| Cache-Control-Beachtung | `proxy_cache_valid` mit Origin-Header-Durchleitung |
| Cache-Status-Berichterstattung | `add_header X-Cache-Status $upstream_cache_status` |
| Health-Endpunkt | `/health`-Location, die 200 OK zurГјckgibt |

### Cache-Verhalten

- Erster Request an einen beliebigen Pfad: `X-Cache-Status: MISS` (vom Origin geholt, jetzt gecached)
- Nachfolgende identische Requests: `X-Cache-Status: HIT` (aus dem Disk-Cache ausgeliefert)
- Cache-Key: `$scheme$host$request_uri` (Schema + Hostname + vollstГӨndiger Pfad + Query-String)
- Cache-TTL: 10 Minuten fГјr 200/301/302, 1 Minute fГјr 404
- Stale-Serving: liefert gecachte Inhalte bei Origin-Fehlern (500/502/503/504)
