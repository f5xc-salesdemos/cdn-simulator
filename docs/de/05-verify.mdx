Führen Sie nach dem Deployment diese Prüfungen durch, um zu bestätigen, dass der CDN-Edge-Knoten korrekt funktioniert. Warten Sie 2-3 Minuten nach `terraform apply`, bis cloud-init die Installation von NGINX abgeschlossen hat.

## Gesundheitsprüfung

Überprüfen Sie, ob NGINX läuft und antwortet:

```bash
curl -s "http://<PUBLIC_IP>/health" | jq .
```

Erwartete Ausgabe:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

Bei Verwendung von Terraform-Ausgaben:

```bash
curl -s "$(terraform output -raw health_check_url)" | jq .
```

## Cache Miss (Erste Anfrage)

Die erste Anfrage an einen beliebigen Pfad wird ein Cache Miss sein — der Edge-Knoten ruft die Daten vom Origin ab:

```bash
curl -I "http://<PUBLIC_IP>/"
```

Achten Sie auf diese Header in der Antwort:

```
X-Cache-Status: MISS
X-CDN-Edge: cdn-simulator
```

`MISS` bedeutet, dass der Inhalt nicht im Cache war und vom Origin-Server abgerufen wurde.

## Cache Hit (Nachfolgende Anfrage)

Wiederholen Sie die gleiche Anfrage sofort:

```bash
curl -I "http://<PUBLIC_IP>/"
```

Erwartete Header:

```
X-Cache-Status: HIT
X-CDN-Edge: cdn-simulator
```

`HIT` bestätigt, dass die Antwort aus dem NGINX-Festplatten-Cache bereitgestellt wurde, ohne den Origin zu kontaktieren.

## Cache-Verzeichnis-Inspektion

Verbinden Sie sich per SSH mit der VM, um zu überprüfen, ob zwischengespeicherte Inhalte auf der Festplatte vorhanden sind:

```bash
ssh azureuser@<PUBLIC_IP>

# Prüfen, ob das Cache-Verzeichnis Inhalte enthält
sudo find /var/cache/nginx/cdn -type f | head -20

# Cache-Größe prüfen
sudo du -sh /var/cache/nginx/cdn
```

## NGINX-Status

Überprüfen Sie, ob NGINX läuft und die Konfiguration gültig ist:

```bash
ssh azureuser@<PUBLIC_IP>

# Dienststatus prüfen
sudo systemctl status nginx

# Konfiguration validieren
sudo nginx -t

# Aktive Verbindungen anzeigen (wenn stub_status aktiviert ist)
curl -s http://localhost/health
```

## Origin-Konnektivität

Überprüfen Sie, ob der Edge-Knoten den Origin-Server erreichen kann:

```bash
ssh azureuser@<PUBLIC_IP>

# Konnektivität zum Origin testen (ersetzen Sie durch Ihre Origin-URL)
curl -I "https://your-origin.example.com/"
```

Falls dies fehlschlägt, prüfen Sie:

1. Die Variable `origin_server` ist in der Terraform-Konfiguration korrekt
2. Der Origin-Server erlaubt eingehende Verbindungen von der öffentlichen IP des Edge-Knotens
3. Die DNS-Auflösung funktioniert von der VM aus (`nslookup your-origin.example.com`)

## Cloud-Init-Fortschritt

Überwachen Sie den Provisionierungsfortschritt über das Cloud-Init-Log:

```bash
ssh azureuser@<PUBLIC_IP> "tail -f /var/log/cloud-init-progress.log"
```

Erwartete Phasen: `[init]`, `[nic]` (dynamische NIC-Erkennung), `[complete]`.

Falls cloud-init Fehler meldet:

```bash
ssh azureuser@<PUBLIC_IP> "sudo cloud-init status --long"
```

## End-to-End-Test

Führen Sie einen vollständigen Anfragezyklus durch und überprüfen Sie die Antwortkette:

```bash
# Erste Anfrage — MISS (wird vom Origin abgerufen)
echo "=== Request 1 (expect MISS) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"

# Zweite Anfrage — HIT (wird aus dem Cache bereitgestellt)
echo "=== Request 2 (expect HIT) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"
```

Erwartete Ausgabe:

```
=== Request 1 (expect MISS) ===
HTTP 200 | Cache: MISS
=== Request 2 (expect HIT) ===
HTTP 200 | Cache: HIT
```

## Vendor-Header-Verifizierung

Überprüfen Sie, ob alle CDN-Vendor-Header eingefügt werden, indem Sie den Pfad `/headers` aufrufen (bei Verwendung von httpbin.org als Origin):

```bash
curl -s "http://<PUBLIC_IP>/headers" | python3 -m json.tool
```

Die erwartete Antwort enthält Header von allen fünf Anbietern:

```json
{
    "headers": {
        "True-Client-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Connecting-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Ipcountry": "US",
        "Cf-Ray": "<request_id>-SJC",
        "Cf-Bot-Score": "85",
        "Cf-Ja3-Hash": "e7d705a3286e19ea42f587b344ee6865",
        "Cloudfront-Viewer-Country": "US",
        "Cloudfront-Viewer-City": "San Jose",
        "Cloudfront-Is-Desktop-Viewer": "true",

        "X-Geo-Country-Code": "US"
    }

```

### Geräteerkennung testen

Testen Sie mit einem mobilen User-Agent, um zu überprüfen, ob sich die Geräteerkennungs-Header ändern:

```bash
curl -s "http://<PUBLIC_IP>/headers?device=mobile" \
  -H "User-Agent: Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 like Mac OS X)" \
  | python3 -m json.tool | grep -E "Is-Mobile|Is-Desktop|is_mobile"
```

Erwartet:

```
"Cloudfront-Is-Desktop-Viewer": "false",
"Cloudfront-Is-Mobile-Viewer": "true",
```

## Fehlerbehebung

### NGINX läuft nicht

```bash
ssh azureuser@<PUBLIC_IP>
sudo systemctl status nginx
sudo journalctl -u nginx --no-pager -n 50
```

### Cloud-init noch nicht abgeschlossen

Nach `terraform apply` benötigt cloud-init 2-3 Minuten. Fortschritt prüfen:

```bash
ssh azureuser@<PUBLIC_IP>
sudo cloud-init status
sudo tail -f /var/log/cloud-init-output.log
```

### Cache zeigt immer MISS an

- Überprüfen Sie, ob der Origin cachefähige Antwortcodes zurückgibt (200, 301, 302)
- Prüfen Sie, ob der Origin keine `Cache-Control: no-cache`- oder `no-store`-Header sendet
- Prüfen Sie die NGINX-Fehlerprotokolle: `sudo tail -f /var/log/nginx/error.log`

### Verbindung abgelehnt auf Port 80/443

- Überprüfen Sie, ob die NSG-Regeln angewendet sind: `az network nsg rule list --nsg-name "$(terraform output -raw nsg_name)" --resource-group "$(terraform output -raw resource_group_name)" -o table`
- Überprüfen Sie, ob NGINX lauscht: `ssh azureuser@<PUBLIC_IP> "sudo ss -tlnp | grep nginx"`
