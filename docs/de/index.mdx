## Was dies bereitstellt

- **[Einzelner Edge-Knoten](01-overview/)** — Eine Ubuntu 24.04 VM mit NGINX als Caching-Reverse-Proxy, die minimal funktionsfähige CDN-Edge
- **[Terraform-Bereitstellung](03-deploy/)** — Vollständige Azure-Infrastruktur als Code: VNet, PIP, NSG und VM mit cloud-init-Provisionierung
- **[67+ CDN-Anbieter-Header](04-nginx-config/)** — Simuliert Request-Header von Akamai, Cloudflare, CloudFront, Fastly und Azure Front Door, einschließlich True-Client-IP, CF-Connecting-IP, Geolokalisierung, Geräteerkennung, JA3/JA4-Fingerprints und Bot-Scores
- **[Origin-Integration](06-integrate/)** — Konfigurieren Sie die Edge so, dass Cache-Misses an einen F5 XC HTTP Load Balancer als Origin weitergeleitet werden
- **[Cache-Verifizierung](05-verify/)** — Validieren Sie HIT/MISS-Verhalten mit X-Cache-Status-Headern und NGINX-Cache-Inspektion

## Kurzreferenz

Nach der Bereitstellung stellt der CDN-Edge-Knoten folgende Endpunkte bereit:

| Endpunkt | Methode | Zweck |
|---|---|---|
| `http://<PUBLIC_IP>/health` | GET | Health Check — gibt JSON mit Komponentenstatus und Anbieterprofilen zurück |
| `http://<PUBLIC_IP>/` | GET | CDN-Proxy — cached und leitet Anfragen an den Origin-Server weiter |
| `http://<PUBLIC_IP>/<any-path>` | GET | CDN-Proxy — alle Pfade werden mit Caching und Anbieter-Headern proxied |

Antwort-Header, die jeder proxied Antwort hinzugefügt werden:

| Header | Beispiel | Zweck |
|---|---|---|
| `X-Cache-Status` | `HIT` oder `MISS` | Ob die Antwort aus dem Cache bedient wurde |
| `X-CDN-Edge` | `cdn-simulator` | Identifiziert diesen Edge-Knoten |
| `X-CDN-POP` | `SJC` | Simulierter Point-of-Presence-Code |
| `X-Served-By` | `cache-sjc3120-SJC` | Simulierter Cache-Knoten-Bezeichner |
| `X-Request-ID` | UUID | Eindeutiger Bezeichner pro Anfrage |
