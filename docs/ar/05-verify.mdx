بعد النشر، قم بتنفيذ هذه الفحوصات للتأكد من أن عقدة CDN الطرفية تعمل بشكل صحيح. انتظر 2-3 دقائق بعد تنفيذ `terraform apply` حتى ينتهي cloud-init من تثبيت NGINX.

## فحص الصحة

تحقق من أن NGINX يعمل ويستجيب:

```bash
curl -s "http://<PUBLIC_IP>/health" | jq .
```

المخرجات المتوقعة:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

إذا كنت تستخدم مخرجات Terraform:

```bash
curl -s "$(terraform output -raw health_check_url)" | jq .
```

## فقدان ذاكرة التخزين المؤقت (الطلب الأول)

سيكون الطلب الأول لأي مسار فقدان في ذاكرة التخزين المؤقت — حيث تجلب العقدة الطرفية المحتوى من خادم المصدر:

```bash
curl -I "http://<PUBLIC_IP>/"
```

ابحث عن هذه الترويسات في الاستجابة:

```
X-Cache-Status: MISS
X-CDN-Edge: cdn-simulator
```

`MISS` يعني أن المحتوى لم يكن موجوداً في ذاكرة التخزين المؤقت وتم جلبه من خادم المصدر.

## إصابة ذاكرة التخزين المؤقت (الطلب اللاحق)

كرر نفس الطلب فوراً:

```bash
curl -I "http://<PUBLIC_IP>/"
```

الترويسات المتوقعة:

```
X-Cache-Status: HIT
X-CDN-Edge: cdn-simulator
```

`HIT` يؤكد أن الاستجابة تم تقديمها من ذاكرة التخزين المؤقت على قرص NGINX دون الاتصال بخادم المصدر.

## فحص دليل ذاكرة التخزين المؤقت

اتصل بالجهاز الافتراضي عبر SSH للتحقق من وجود المحتوى المخزن مؤقتاً على القرص:

```bash
ssh azureuser@<PUBLIC_IP>

# تحقق من أن دليل التخزين المؤقت يحتوي على محتوى
sudo find /var/cache/nginx/cdn -type f | head -20

# تحقق من حجم التخزين المؤقت
sudo du -sh /var/cache/nginx/cdn
```

## حالة NGINX

تحقق من أن NGINX يعمل وأن التكوين صالح:

```bash
ssh azureuser@<PUBLIC_IP>

# تحقق من حالة الخدمة
sudo systemctl status nginx

# تحقق من صحة التكوين
sudo nginx -t

# عرض الاتصالات النشطة (إذا كان stub_status مُفعلاً)
curl -s http://localhost/health
```

## الاتصال بخادم المصدر

تحقق من أن العقدة الطرفية يمكنها الوصول إلى خادم المصدر:

```bash
ssh azureuser@<PUBLIC_IP>

# اختبار الاتصال بخادم المصدر (استبدل بعنوان URL الخاص بخادم المصدر)
curl -I "https://your-origin.example.com/"
```

إذا فشل هذا الاختبار، تحقق من:

1. أن متغير `origin_server` صحيح في تكوين Terraform
2. أن خادم المصدر يسمح بالاتصالات الواردة من عنوان IP العام للعقدة الطرفية
3. أن تحليل DNS يعمل من الجهاز الافتراضي (`nslookup your-origin.example.com`)

## تقدم Cloud-Init

راقب تقدم التزويد عبر سجل cloud-init:

```bash
ssh azureuser@<PUBLIC_IP> "tail -f /var/log/cloud-init-progress.log"
```

المراحل المتوقعة: `[init]`، `[nic]` (الكشف الديناميكي عن واجهة الشبكة)، `[complete]`.

إذا أبلغ cloud-init عن أخطاء:

```bash
ssh azureuser@<PUBLIC_IP> "sudo cloud-init status --long"
```

## الاختبار الشامل

قم بتنفيذ دورة طلب كاملة وتحقق من سلسلة الاستجابة:

```bash
# الطلب الأول — MISS (جلب من خادم المصدر)
echo "=== Request 1 (expect MISS) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"

# الطلب الثاني — HIT (تقديم من ذاكرة التخزين المؤقت)
echo "=== Request 2 (expect HIT) ==="
curl -s -o /dev/null -w "HTTP %{http_code} | Cache: %{header:X-Cache-Status}\n" "http://<PUBLIC_IP>/test-path"
```

المخرجات المتوقعة:

```
=== Request 1 (expect MISS) ===
HTTP 200 | Cache: MISS
=== Request 2 (expect HIT) ===
HTTP 200 | Cache: HIT
```

## التحقق من ترويسات مزودي CDN

تحقق من أن جميع ترويسات مزودي CDN يتم حقنها عن طريق طلب المسار `/headers` (عند استخدام httpbin.org كخادم مصدر):

```bash
curl -s "http://<PUBLIC_IP>/headers" | python3 -m json.tool
```

الاستجابة المتوقعة تتضمن ترويسات من جميع المزودين الخمسة:

```json
{
    "headers": {
        "True-Client-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Connecting-Ip": "<YOUR_CLIENT_IP>",
        "Cf-Ipcountry": "US",
        "Cf-Ray": "<request_id>-SJC",
        "Cf-Bot-Score": "85",
        "Cf-Ja3-Hash": "e7d705a3286e19ea42f587b344ee6865",
        "Cloudfront-Viewer-Country": "US",
        "Cloudfront-Viewer-City": "San Jose",
        "Cloudfront-Is-Desktop-Viewer": "true",
        "Cloudfront-Is-Mobile-Viewer": "false",
        "Fastly-Client-Ip": "<YOUR_CLIENT_IP>",
        "X-Akamai-Edgescape": "georegion=263,country_code=US,...",
        "X-Azure-Clientip": "<YOUR_CLIENT_IP>",
        "X-Azure-Fdid": "a0a0a0a0-bbbb-cccc-dddd-e1e1e1e1e1e1",
        "X-Geo-Country-Code": "US"
    }
}
```

### اختبار كشف نوع الجهاز

اختبر باستخدام وكيل مستخدم للأجهزة المحمولة للتحقق من تغير ترويسات كشف نوع الجهاز:

```bash
curl -s "http://<PUBLIC_IP>/headers?device=mobile" \
  -H "User-Agent: Mozilla/5.0 (iPhone; CPU iPhone OS 17_0 like Mac OS X)" \
  | python3 -m json.tool | grep -E "Is-Mobile|Is-Desktop|is_mobile"
```

المتوقع:

```
"Cloudfront-Is-Desktop-Viewer": "false",
"Cloudfront-Is-Mobile-Viewer": "true",
```

## استكشاف الأخطاء وإصلاحها

### NGINX لا يعمل

```bash
ssh azureuser@<PUBLIC_IP>
sudo systemctl status nginx
sudo journalctl -u nginx --no-pager -n 50
```

### لم ينتهِ cloud-init بعد

بعد تنفيذ `terraform apply`، يستغرق cloud-init من 2-3 دقائق. تحقق من التقدم:

```bash
ssh azureuser@<PUBLIC_IP>
sudo cloud-init status
sudo tail -f /var/log/cloud-init-output.log
```

### ذاكرة التخزين المؤقت تُظهر MISS دائماً

- تحقق من أن خادم المصدر يُرجع رموز استجابة قابلة للتخزين المؤقت (200، 301، 302)
- تحقق من أن خادم المصدر لا يُرسل ترويسات `Cache-Control: no-cache` أو `no-store`
- تحقق من سجلات أخطاء NGINX: `sudo tail -f /var/log/nginx/error.log`

### رفض الاتصال على المنفذ 80/443

- تحقق من تطبيق قواعد NSG: `az network nsg rule list --nsg-name "$(terraform output -raw nsg_name)" --resource-group "$(terraform output -raw resource_group_name)" -o table`
- تحقق من أن NGINX يستمع: `ssh azureuser@<PUBLIC_IP> "sudo ss -tlnp | grep nginx"`
