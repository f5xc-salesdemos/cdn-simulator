## الغرض

يحاكي هذا المكون عقدة حافة CDN لبيئات المختبرات والعروض التوضيحية. وهو يمثل الدور الذي يلعبه مزودون مثل Akamai وCloudflare وAmazon CloudFront أو Fastly في البنية الشبكية للعميل — وهو طبقة التخزين المؤقت الأقرب للمستخدمين النهائيين والتي تقع أمام خادم الأصل.

في بنيات الإنتاج متعددة المزودين، يقوم العملاء عادةً بدمج CDN من طرف ثالث مع F5 Distributed Cloud:

```
End User → CDN Edge (Akamai/Cloudflare/etc.) → F5 XC HTTP LB → Origin App
```

يستبدل هذا المحاكي CDN التجاري بعقدة حافة مبنية على NGINX بحيث يمكن عرض واختبار التكامل دون الحاجة إلى عقود مع مزودين أو بنية إنتاجية تحتية.

## البنية المعمارية

```
┌─────────┐     ┌──────────────────────┐     ┌─────────────────┐     ┌────────────┐
│  Client  │────▶│  CDN Edge (NGINX)    │────▶│  F5 XC HTTP LB  │────▶│ Origin App │
│          │     │  Ubuntu 24.04 Azure  │     │  (Origin Server) │     │            │
└─────────┘     │  - TLS termination   │     └─────────────────┘     └────────────┘
                │  - Disk-based cache  │
                │  - X-Cache-Status    │
                └──────────────────────┘
```

عقدة حافة NGINX:

- **تنهي TLS** عند الحافة (شهادة موقعة ذاتياً أو Let's Encrypt)
- **تخزن الاستجابات مؤقتاً** على القرص باستخدام `proxy_cache_path`
- **تعيد توجيه طلبات عدم التخزين المؤقت** إلى خادم أصل قابل للتكوين (عنوان VIP لموزع الحمل HTTP في F5 XC)
- **تُبلغ عن حالة التخزين المؤقت** عبر ترويسة الاستجابة `X-Cache-Status` (`HIT`، `MISS`، `BYPASS`، `EXPIRED`)

## ما يحاكيه هذا المكون

| وظيفة CDN | تنفيذ NGINX |
|---|---|
| التخزين المؤقت عند الحافة | `proxy_cache` مع تخزين على القرص |
| توليد مفتاح التخزين المؤقت | `proxy_cache_key` بناءً على المخطط والمضيف ومعرف URI |
| سحب من الأصل | `proxy_pass` إلى موزع حمل HTTP في F5 XC |
| إنهاء TLS | توجيه `ssl_certificate` في NGINX |
| احترام Cache-Control | `proxy_cache_valid` مع تمرير ترويسات الأصل |
| الإبلاغ عن حالة التخزين المؤقت | `add_header X-Cache-Status $upstream_cache_status` |
| نقطة نهاية الصحة | موقع `/health` يُرجع 200 OK |

## نقاط النهاية وسلوك الطلب/الاستجابة

### فحص الصحة

```
GET /health
```

الاستجابة (200 OK، `Content-Type: application/json`):

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": ["akamai", "cloudflare", "cloudfront", "fastly", "azure-front-door"]
}
```

### وكيل CDN (جميع المسارات الأخرى)

```
GET /<any-path>
```

ترويسات الطلب المُدرجة نحو الأصل (أكثر من 67 ترويسة من 5 مزودين):

| الفئة | الترويسات المُضافة |
|---|---|
| عنوان IP للعميل | `True-Client-IP`، `CF-Connecting-IP`، `Fastly-Client-IP`، `X-Azure-ClientIP`، `CloudFront-Viewer-Address`، `X-Forwarded-For`، `X-Real-IP` |
| الموقع الجغرافي | `X-Akamai-Edgescape` (مركّب)، `CF-IPCountry`، `cf-ipcity`، `cf-iplatitude/longitude`، `CloudFront-Viewer-Country/City/Latitude/Longitude`، `X-Geo-Country-Code/City/Region` |
| كشف الجهاز | `CloudFront-Is-Mobile-Viewer`، `CloudFront-Is-Desktop-Viewer`، `CloudFront-Is-Tablet-Viewer`، `X-Akamai-Device-Characteristics` |
| TLS/البصمة | `CloudFront-Viewer-TLS`، `cf-ja3-hash`، `cf-ja4`، `CloudFront-Viewer-JA3-Fingerprint` |
| كشف الروبوتات | `cf-bot-score` (85 = على الأرجح بشري)، `cf-verified-bot` |
| تتبع الطلبات | `Cf-Ray`، `X-Akamai-Request-ID`، `X-Amz-Cf-Id`، `X-Azure-Ref` |
| هوية الحافة | `X-CDN-Edge`، `X-CDN-POP`، `X-Served-By`، `Fastly-FF`، `X-Azure-FDID` |
| قياسية | `Via`، `Forwarded`، `CDN-Loop`، `X-Forwarded-Proto/Host/Port` |

ترويسات الاستجابة المُضافة لكل استجابة مُوكّلة:

| الترويسة | القيم | الغرض |
|---|---|---|
| `X-Cache-Status` | `HIT`، `MISS`، `BYPASS`، `EXPIRED`، `STALE` | سلوك التخزين المؤقت لهذا الطلب |
| `X-CDN-Edge` | `cdn-simulator` | تعريف عقدة الحافة هذه |
| `X-CDN-POP` | `SJC` | رمز IATA لنقطة التواجد المُحاكاة |
| `X-Served-By` | `cache-sjc3120-SJC` | عقدة تخزين مؤقت مُحاكاة بتنسيق Fastly |
| `X-Request-ID` | UUID (لكل طلب) | معرف فريد للطلب |

### سلوك التخزين المؤقت

- الطلب الأول لأي مسار: `X-Cache-Status: MISS` (تم جلبه من الأصل، وأصبح مخزناً مؤقتاً الآن)
- الطلبات المتطابقة اللاحقة: `X-Cache-Status: HIT` (يُقدم من التخزين المؤقت على القرص)
- مفتاح التخزين المؤقت: `$scheme$host$request_uri` (المخطط + اسم المضيف + المسار الكامل + سلسلة الاستعلام)
- مدة صلاحية التخزين المؤقت: 10 دقائق للحالات 200/301/302، دقيقة واحدة للحالة 404
- التقديم من المخزن القديم: يُرجع المحتوى المخزن مؤقتاً عند حدوث أخطاء في الأصل (500/502/503/504)

### الوصول إلى الجهاز الافتراضي

| طريقة الوصول | الأمر/المسار |
|---|---|
| SSH | `ssh azureuser@<PUBLIC_IP>` |
| تكوين NGINX | `/etc/nginx/conf.d/cdn-edge.conf` |
| سجلات NGINX | `/var/log/nginx/access.log` و `/var/log/nginx/error.log` |
| دليل التخزين المؤقت | `/var/cache/nginx/cdn/` |
| سجل Cloud-init | `/var/log/cloud-init-output.log` |

## التصميم المعياري للمكونات

هذا جزء واحد من بيئة مختبر أكبر. كل مكون مستقل بذاته ويُنشر بشكل مستقل:

- **هذا المكون** يوفر حافة CDN (NGINX على جهاز Azure الافتراضي)
- **المكونات الأخرى** توفر تطبيق الأصل، وتكوين F5 XC، وDNS، وسياسات WAF، وغيرها

يضيف المشغل البشري المكونات واحداً تلو الآخر. وثائق كل مكون مكتوبة بحيث يمكن لمساعد الذكاء الاصطناعي قراءتها ونشر البنية التحتية بشكل مستقل.

## لماذا NGINX

تم اختيار NGINX كمحرك لمحاكاة CDN للأسباب التالية:

1. **منتج F5** — استحوذت F5 على شركة NGINX Inc. في عام 2019؛ وهو جزء من محفظة منتجات F5
2. **مثبت في الصناعة** — شغّلت Cloudflare شبكة CDN الخاصة بها بالكامل على NGINX لأكثر من عقد قبل الانتقال إلى Pingora؛ وتستخدم Netflix NGINX لشبكة CDN الخاصة بها Open Connect
3. **عملية واحدة** — يتعامل مع إنهاء TLS والتخزين المؤقت في ملف ثنائي واحد، على عكس Varnish الذي يتطلب وكيل TLS منفصلاً
4. **نشر بسيط** — `apt install nginx` على Ubuntu 24.04، وتوجيهان يُفعّلان التخزين المؤقت
5. **موثق جيداً** — وثائق رسمية شاملة لتخزين المحتوى المؤقت وتكوين الوكيل العكسي
