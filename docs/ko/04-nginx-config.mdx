cloud-init 프로비저닝은 완전히 성능 최적화된 NGINX CDN 엣지 구성을 배포합니다. 이 페이지는 커널 튜닝부터 캐시 동작, 벤더 헤더 주입까지 모든 구성 레이어를 문서화합니다. 모든 설정은 172,540 req/s 피크의 48시간 연속 부하 테스트로 검증되었습니다.

## 구성 파일

| 파일 | 용도 |
|---|---|
| `/etc/sysctl.d/99-cdn-tuning.conf` | Linux 커널 네트워크 튜닝 |
| `/etc/systemd/system/nginx.service.d/override.conf` | NGINX 파일 디스크립터 제한 |
| `/etc/security/limits.d/99-nginx.conf` | www-data 사용자의 OS 수준 제한 |
| `/etc/nginx/nginx.conf` | NGINX 메인 구성 (워커, 버퍼, gzip, 로깅) |
| `/etc/nginx/conf.d/cdn-edge.conf` | CDN 프록시 구성 (캐시, 업스트림, 헤더) |

## NGINX 메인 구성

### 워커 및 연결

```nginx
worker_processes auto;           # 4 workers on D4s_v5 (1 per vCPU)
worker_rlimit_nofile 65535;      # Per-worker file descriptor limit

events {
    use epoll;                   # Linux-optimized event model
    worker_connections 8192;     # 4 workers x 8192 = 32,768 max concurrent
    multi_accept on;             # Accept all pending connections per event loop
    accept_mutex off;            # Not needed with epoll + reuseport
}
```

`/etc/systemd/system/nginx.service.d/override.conf`의 Systemd 오버라이드가 `LimitNOFILE=65535`를 일치하도록 설정합니다.

### 프록시 버퍼

```nginx
proxy_buffering on;
proxy_buffer_size 16k;           # Header buffer (handles large CDN headers)
proxy_buffers 64 16k;            # 1 MB per connection (64 x 16k)
proxy_busy_buffers_size 256k;    # Can send 256k to client while still reading
```

256k busy 버퍼 크기는 매우 중요합니다 — 가장 큰 캐시된 응답보다 커야 합니다 (Juice Shop은 75 KB). 원래 64k 설정은 부하 시 직렬화를 유발했습니다.

### Gzip 압축

```nginx
gzip on;
gzip_comp_level 4;               # Balance: ~85% of max compression at ~40% CPU
gzip_min_length 256;             # Skip tiny responses (gzip header overhead)
gzip_vary on;                    # Vary: Accept-Encoding for correct caching
gzip_proxied any;                # Compress all proxied responses
gzip_types text/plain text/css text/javascript text/xml
           application/json application/javascript application/xml
           application/xml+rss application/atom+xml
           application/ld+json application/manifest+json
           image/svg+xml;
```

### 열린 파일 캐시

```nginx
open_file_cache max=200000 inactive=20s;
open_file_cache_valid 30s;
open_file_cache_min_uses 2;
open_file_cache_errors on;
```

캐시된 객체의 파일 디스크립터와 메타데이터를 캐시하여 핫 파일에 대한 `stat()` 및 `open()` 시스콜을 제거합니다.

### 클라이언트 Keepalive

```nginx
keepalive_timeout 65;
keepalive_requests 100000;       # Was 1000 — caused connection recycling at 90K req/s
```

90K req/s에서 `keepalive_requests 1000`으로 설정 시, 연결이 11초마다 재활용되어 TIME_WAIT 소켓을 생성했습니다. 100,000으로 증가시키자 이 문제가 완전히 해결되었습니다.

### 접근 로깅

```nginx
log_format cdn '$remote_addr [$time_local] "$request" $status $body_bytes_sent $upstream_cache_status $request_time';
access_log /var/log/nginx/access.log cdn buffer=256k flush=5s;
```

`cdn` 로그 형식은 지연 시간 분석을 위한 `$upstream_cache_status` (HIT/MISS)와 `$request_time`을 포함합니다. 버퍼링된 로깅(256k/5s 플러시)은 높은 부하에서 I/O 오버헤드를 줄입니다.

## 업스트림 Keepalive

```nginx
upstream origin_backend {
    server 20.12.78.159:80;
    keepalive 256;               # Persistent connections per worker to origin
    keepalive_timeout 60s;
    keepalive_requests 1000;
}
```

4개 워커 x 256 keepalive = 오리진에 대한 1,024개의 웜 연결. `proxy_http_version 1.1` 및 `proxy_set_header Connection ""`과 결합하면 모든 캐시 미스에서 TCP 핸드셰이크 오버헤드가 제거됩니다.

## 캐시 구성

### 캐시 경로

```nginx
proxy_cache_path /var/cache/nginx/cdn
                 levels=1:2
                 keys_zone=cdn_cache:32m
                 max_size=25g
                 inactive=24h
                 use_temp_path=off;
```

| 파라미터 | 값 | 용도 |
|---|---|---|
| `keys_zone=cdn_cache:32m` | 32 MB 공유 메모리 | ~256,000개의 캐시 키와 메타데이터 저장 |
| `max_size=25g` | 25 GB 디스크 제한 | 30 GB OS 디스크의 대부분을 사용; 제한 시 LRU 제거 |
| `inactive=24h` | 24시간 비활성 타임아웃 | 24시간 동안 접근되지 않은 콘텐츠 제거 (TTL과 동일하지 않음) |
| `use_temp_path=off` | 캐시 디렉토리에 직접 쓰기 | 크로스 파일시스템 파일 이동 방지 |

### 캐시 동작

```nginx
proxy_cache cdn_cache;
proxy_cache_valid 200 301 302 4h;
proxy_cache_valid 404 1m;
proxy_cache_key "$scheme$host$request_uri";
proxy_cache_lock on;
proxy_cache_lock_age 3s;
proxy_cache_lock_timeout 3s;
proxy_cache_background_update on;
proxy_cache_use_stale updating error timeout http_500 http_502 http_503 http_504;
proxy_ignore_headers Set-Cookie Cache-Control Expires Vary;

proxy_hide_header X-Cache-Status;
proxy_hide_header Vary;
```

| 디렉티브 | 값 | 용도 |
|---|---|---|
| `proxy_cache_valid 200 301 302 4h` | 4시간 TTL | 캐시된 콘텐츠가 4시간 동안 유효 (갱신 파동 처리량 저하를 줄이기 위해 증가됨) |
| `proxy_cache_lock on` | 썬더링 허드 방지 | 캐시되지 않은 URL당 1개의 요청만 오리진으로 전달; 나머지는 캐시 채움 대기 |
| `proxy_cache_lock_age 3s` | 잠금 타임아웃 | 첫 번째 요청이 3초 내에 완료되지 않으면 다른 요청 허용 |
| `proxy_cache_background_update on` | 무지연 갱신 | 백그라운드에서 갱신하는 동안 오래된 콘텐츠를 즉시 제공 |
| `proxy_cache_use_stale` | 복원력 | 오리진 오류(500/502/503/504) 또는 업데이트 중에 오래된 콘텐츠 제공 |
| `proxy_ignore_headers` | 강제 캐싱 | 오리진의 Set-Cookie, Cache-Control, Expires, Vary 무시 — CDN이 TTL과 Vary 동작을 결정 (Juice Shop은 `max-age=0`과 삼중 `Vary` 헤더를 전송하여 효과적인 캐싱을 방해함) |
| `proxy_hide_header X-Cache-Status` | 오리진 헤더 제거 | 오리진 NGINX가 자체 X-Cache-Status를 추가 — CDN의 캐시 상태만 표시되도록 제거 |
| `proxy_hide_header Vary` | 캐시 단편화 방지 | 오리진이 여러 `Vary: Accept-Encoding` 헤더를 전송. 이를 제거하면 `Accept-Encoding` 순열 간의 캐시 키 단편화를 방지. NGINX의 `gzip_vary on`이 올바른 단일 `Vary` 헤더를 자동으로 추가 |

### 프록시 타임아웃

```nginx
proxy_read_timeout 30s;
proxy_connect_timeout 10s;
proxy_send_timeout 15s;
```

부하 시 오리진에 더 많은 응답 시간을 제공하면서도 연결 문제 시 빠르게 실패합니다.

### 서버 블록

```nginx
server {
    listen 80 reuseport;        # Kernel distributes connections across all 4 workers
    server_name _;
}
```

`reuseport`는 `SO_REUSEPORT`를 활성화합니다 — 커널이 수신 연결을 워커 프로세스에 직접 분배하여 accept 뮤텍스 경합을 제거합니다.

## CDN 벤더 헤더

시뮬레이터는 5개 주요 CDN 벤더의 헤더를 모두 동시에 주입합니다. 이를 통해 어떤 CDN이 시뮬레이션되든 관계없이 F5 XC를 모든 벤더의 "Trusted Client IP Header"로 구성하고 현실적인 헤더 페이로드를 확인할 수 있습니다.

### 산업 표준 헤더 (모든 CDN)

| 헤더 | 값 | 용도 |
|---|---|---|
| `X-Forwarded-For` | 클라이언트 IP 체인 | 표준 포워딩 IP |
| `X-Forwarded-Proto` | `http` 또는 `https` | 원본 클라이언트 프로토콜 |
| `X-Forwarded-Host` | 원본 호스트명 | 원본 Host 헤더 |
| `X-Forwarded-Port` | 서버 포트 | 원본 포트 |
| `X-Real-IP` | 클라이언트 IP | 단일 클라이언트 IP (nginx 관례) |
| `Via` | `1.1 cdn-simulator` | 프록시 식별 |
| `Forwarded` | RFC 7239 형식 | 표준화된 포워딩 헤더 |
| `CDN-Loop` | `cdn-simulator` | 루프 감지 |

| `X-Cache-Status` | `HIT`, `MISS`, `EXPIRED`, `STALE`, `UPDATING` | 이 요청의 캐시 동작 |

| `X-CDN-POP` | `SJC` | 시뮬레이션된 POP(Point of Presence) IATA 코드 |
| `X-Served-By` | `cache-sjc3120-SJC` | Fastly 형식의 시뮬레이션된 캐시 노드 |
| `X-Request-ID` | UUID (요청별) | 고유 요청 식별자 |

## 디바이스 감지

시뮬레이터는 NGINX `map` 디렉티브를 사용하여 `User-Agent` 헤더에서 디바이스 유형을 감지합니다:

- **모바일**: iPhone, Android (태블릿 제외), iPod, BlackBerry, Opera Mini, IEMobile 매칭
- **태블릿**: iPad, Android 태블릿, Kindle, PlayBook 매칭
- **데스크톱**: 모바일도 태블릿도 매칭되지 않을 때 기본값

디바이스 유형은 다음에 반영됩니다:
- `CloudFront-Is-Desktop-Viewer` / `CloudFront-Is-Mobile-Viewer` / `CloudFront-Is-Tablet-Viewer`
- `X-Akamai-Device-Characteristics` (`is_mobile`, `is_tablet`, `is_wireless_device` 필드)

## 운영

### 오리진 서버 변경

```bash
ssh azureuser@<PUBLIC_IP>

# Update upstream server
sudo sed -i 's|server .*;|server NEW_HOST:80;|' /etc/nginx/conf.d/cdn-edge.conf

# Clear cache and reload
sudo rm -rf /var/cache/nginx/cdn/*
sudo nginx -t && sudo systemctl reload nginx
```

또는 `terraform.tfvars`에서 `origin_host`를 업데이트하고 `terraform apply`를 실행하여 재프로비저닝합니다.

### 캐시 삭제

```bash
ssh azureuser@<PUBLIC_IP>
sudo rm -rf /var/cache/nginx/cdn/*
sudo systemctl reload nginx
```

### 캐시 통계 확인

```bash
# Object count and disk usage
sudo find /var/cache/nginx/cdn -type f | wc -l
sudo du -sh /var/cache/nginx/cdn

# Recent access log with cache status
tail -20 /var/log/nginx/access.log
```

### 부하 시 모니터링

```bash
# Real-time connections and socket states
ss -s

# NGINX worker CPU usage
top -bn1 | grep nginx

# Upstream keepalive connections
ss -tn state established dst <ORIGIN_IP> | wc -l

# TIME_WAIT socket count
ss -tn state time-wait | wc -l
```
