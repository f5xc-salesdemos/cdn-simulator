Todos os arquivos Terraform estão no diretório [`terraform/`](https://github.com/f5xc-salesdemos/cdn-simulator/tree/main/terraform). Clone o repositório e implante diretamente:

```bash
git clone https://github.com/f5xc-salesdemos/cdn-simulator.git
cd cdn-simulator/terraform
cp terraform.tfvars.example terraform.tfvars
# Edit terraform.tfvars with your Azure subscription ID and origin server
```

## Configuração do Terraform

### Estrutura de Arquivos do Terraform

O diretório terraform contém 9 arquivos seguindo o Padrão de Recursos de Demo:

- `versions.tf` — Restrições de versão do Terraform e do provider (azurerm ~> 4.0, azuread ~> 3.0)
- `providers.tf` — Configuração dos providers Azure RM e Azure AD
- `data.tf` — Fontes de dados do Azure AD para resolução automática do implantador
- `locals.tf` — Resolução do implantador, nomenclatura de recursos do Azure Cloud Adoption Framework, tags padrão
- `main.tf` — Grupo de recursos (nomeado `rg-cdn-simulator-{environment}-{deployer}`)
- `variables.tf` — Todas as variáveis de entrada (3 obrigatórias, 8 opcionais)
- `network.tf` — VNet (10.100.0.0/16), sub-rede, IP público, NSG (portas 22/80/443), NIC
- `vm.tf` — VM Ubuntu 24.04 com cloud-init via templatefile()
- `outputs.tf` — 17 saídas (15 padrão + 2 específicas do componente)

`variables.tf` define 11 variáveis de entrada organizadas nas seções General, Compute e Component-Specific. O identificador `deployer` é resolvido automaticamente a partir da sua conta do Azure AD — você só precisa definir `subscription_id`, `origin_server` e `origin_host`:

```hcl file=../_data/variables.tf
```

### Infraestrutura de Rede

`network.tf` cria a VNet, sub-rede, IP público, NSG (portas 22/80/443) e NIC:

```hcl file=../_data/network.tf
```

### Máquina Virtual com Cloud-Init

`vm.tf` cria a VM Ubuntu 24.04. O caminho da chave pública SSH é expandido via `pathexpand()` para tratar `~`. O template cloud-init recebe as variáveis `origin_server` e `origin_host`:

```hcl file=../_data/vm.tf
```

### Provisionamento com Cloud-Init

`cloud-init.yaml` provisiona a VM com ajuste de kernel, limites do systemd, NGINX com configuração de desempenho otimizado, zona de chaves de cache de 128 MB, pool de keepalive upstream, compressão gzip e mais de 67 cabeçalhos de fornecedores CDN. Uma biblioteca auxiliar compartilhada (`/usr/local/lib/cloud-init-helpers.sh`) fornece lógica de nova tentativa e registro de progresso em `/var/log/cloud-init-progress.log`.

O cloud-init utiliza variáveis de template do Terraform: `${origin_server}` e `${origin_host}` para a configuração upstream. Variáveis NGINX como `${request_id}` são escapadas como `$${request_id}` no templatefile do Terraform.

```yaml file=../_data/cloud-init.yaml
```

### Saídas

`outputs.tf` expõe 17 saídas seguindo o Padrão de Recursos de Demo — 15 saídas padrão compartilhadas por todos os recursos de demo (`deployer`, `public_ip`, `private_ip`, `ssh_command`, `resource_group_name`, `vm_name`, `nsg_name`, `vnet_name`, `subnet_id`, `component`, `environment`, `resource_group_id`, `vm_id`, `nsg_id`, `location`) mais 2 saídas específicas do componente (`edge_url`, `health_check_url`):

```hcl file=../_data/outputs.tf
```

### Arquivo de Variáveis de Exemplo

Copie `terraform.tfvars.example` para `terraform.tfvars` e preencha seus valores. O `.gitignore` exclui `terraform.tfvars` para evitar o commit de credenciais:

```hcl file=../_data/terraform.tfvars.example
```

## Implantação

```bash
# Initialize Terraform
terraform init

# Review the plan
terraform plan

# Apply
terraform apply
```

O Terraform exibe o IP público, o comando SSH e a URL de borda após a implantação bem-sucedida.

## Dimensionamento da VM

As VMs F-series otimizadas para computação são recomendadas para esta carga de trabalho do proxy NGINX com uso intensivo de CPU. A `Standard_F4s_v2` padrão (4 vCPU, 8 GiB) é adequada para uso em laboratório e demo. Substitua a variável `vm_size` para cenários de teste de carga ou benchmarking em produção. O ajuste de kernel do cloud-init e a configuração do NGINX escalam automaticamente com a contagem de vCPU (`worker_processes auto`).

## Pós-Implantação

Após a conclusão do `terraform apply`, aguarde 2 a 3 minutos para que o cloud-init termine de instalar e configurar o NGINX. Verifique o endpoint de saúde:

```bash
curl -s "http://$(terraform output -raw public_ip)/health" | jq .
```

Resposta esperada:

```json
{
  "status": "healthy",
  "component": "cdn-edge",
  "engine": "nginx",
  "vendor_profiles": [
    "akamai",
    "cloudflare",
    "cloudfront",
    "fastly",
    "azure-front-door"
  ]
}
```

## Conexão a partir de Componentes Upstream

O simulador CDN requer um servidor de origem implantado. Utilize as saídas do terraform do servidor de origem para preencher as variáveis obrigatórias:

```bash
cd ../origin-server/terraform
origin_ip=$(terraform output -raw public_ip)

cd ../../cdn-simulator/terraform
cat > terraform.tfvars <<EOF
subscription_id = "your-subscription-id"
origin_server   = "http://${origin_ip}"
origin_host     = "${origin_ip}:80"
EOF
```

| Variável Obrigatória | Origem | Formato |
|---|---|---|
| `origin_server` | Saída `public_ip` do servidor de origem | `http://<ip>` (incluir esquema) |
| `origin_host` | Saída `public_ip` do servidor de origem | `<ip>:80` (sem esquema, incluir porta) |

Prossiga para [Configuração do NGINX](../04-nginx-config/) para opções de personalização ou [Verificar](../05-verify/) para testes de cache.
