## Propósito

Este componente simula um nó de borda CDN para ambientes de laboratório e demonstração. Ele representa o papel que fornecedores como Akamai, Cloudflare, Amazon CloudFront ou Fastly desempenham na arquitetura de rede de um cliente — a camada de cache mais próxima dos usuários finais que fica à frente de um servidor de origem.

Em arquiteturas de produção com múltiplos fornecedores, os clientes comumente combinam um CDN de terceiros com o F5 Distributed Cloud:

```
End User → CDN Edge (Akamai/Cloudflare/etc.) → F5 XC HTTP LB → Origin App
```

Este simulador substitui o CDN comercial por um nó de borda baseado em NGINX para que a integração possa ser demonstrada e testada sem contratos com fornecedores ou infraestrutura de produção.

## Arquitetura

```
┌─────────┐     ┌──────────────────────┐     ┌─────────────────┐     ┌────────────┐
│  Client  │────▶│  CDN Edge (NGINX)    │────▶│  F5 XC HTTP LB  │────▶│ Origin App │
│          │     │  Ubuntu 24.04 Azure  │     │  (Origin Server) │     │            │
└─────────┘     │  - TLS termination   │     └─────────────────┘     └────────────┘
                │  - Disk-based cache  │
                │  - X-Cache-Status    │
                └──────────────────────┘
```

O nó de borda NGINX:

- **Termina TLS** na borda (autoassinado ou Let's Encrypt)
- **Armazena respostas em cache** no disco usando `proxy_cache_path`
- **Encaminha falhas de cache** para um servidor de origem configurável (o VIP do balanceador de carga HTTP do F5 XC)
- **Reporta o status do cache** através do cabeçalho de resposta `X-Cache-Status` (`HIT`, `MISS`, `BYPASS`, `EXPIRED`)

## O Que Isto Simula

| Função CDN | Implementação NGINX |
|---|---|
| Cache de borda | `proxy_cache` com armazenamento baseado em disco |
| Geração de chave de cache | `proxy_cache_key` baseado em esquema, host e URI |
| Busca na origem | `proxy_pass` para o balanceador de carga HTTP do F5 XC |
| Terminação TLS | Diretiva `ssl_certificate` do NGINX |
| Respeito ao Cache-Control | `proxy_cache_valid` com passagem de cabeçalhos da origem |
| Relatório de status do cache | `add_header X-Cache-Status $upstream_cache_status` |
| Endpoint de saúde | Localização `/health` retornando 200 OK |

### Proxy CDN (Todos os Outros Caminhos)

```
GET /<any-path>
```

Cabeçalhos de requisição injetados em direção à origem (67+ cabeçalhos de 5 fornecedores):

| Categoria | Cabeçalhos Adicionados |
|---|---|
| IP do Cliente | `True-Client-IP`, `CF-Connecting-IP`, `Fastly-Client-IP`, `X-Azure-ClientIP`, `CloudFront-Viewer-Address`, `X-Forwarded-For`, `X-Real-IP` |
| Geolocalização | `X-Akamai-Edgescape` (composto), `CF-IPCountry`, `cf-ipcity`, `cf-iplatitude/longitude`, `CloudFront-Viewer-Country/City/Latitude/Longitude`, `X-Geo-Country-Code/City/Region` |
| Detecção de Dispositivo | `CloudFront-Is-Mobile-Viewer`, `CloudFront-Is-Desktop-Viewer`, `CloudFront-Is-Tablet-Viewer`, `X-Akamai-Device-Characteristics` |
| TLS/Fingerprint | `CloudFront-Viewer-TLS`, `cf-ja3-hash`, `cf-ja4`, `CloudFront-Viewer-JA3-Fingerprint` |
| Detecção de Bot | `cf-bot-score` (85 = provavelmente humano), `cf-verified-bot` |
| Rastreamento de Requisição | `Cf-Ray`, `X-Akamai-Request-ID`, `X-Amz-Cf-Id`, `X-Azure-Ref` |
| Identidade da Borda | `X-CDN-Edge`, `X-CDN-POP`, `X-Served-By`, `Fastly-FF`, `X-Azure-FDID` |
| Padrão | `Via`, `Forwarded`, `CDN-Loop`, `X-Forwarded-Proto/Host/Port` |

Cabeçalhos de resposta adicionados a cada resposta proxiada:

| Cabeçalho | Valores | Propósito |
|---|---|---|
| `X-Cache-Status` | `HIT`, `MISS`, `BYPASS`, `EXPIRED`, `STALE` | Comportamento de cache para esta requisição |
| `X-CDN-Edge` | `cdn-simulator` | Identifica este nó de borda |
| `X-CDN-POP` | `SJC` | Código IATA simulado do Ponto de Presença |
| `X-Served-By` | `cache-sjc3120-SJC` | Nó de cache simulado no formato Fastly |
| `X-Request-ID` | UUID (por requisição) | Identificador único de requisição |

### Comportamento do Cache

- Primeira requisição para qualquer caminho: `X-Cache-Status: MISS` (buscado da origem, agora em cache)
- Requisições idênticas subsequentes: `X-Cache-Status: HIT` (servido do cache em disco)
- Chave de cache: `$scheme$host$request_uri` (esquema + hostname + caminho completo + query string)
- TTL do cache: 10 minutos para 200/301/302, 1 minuto para 404
- Servir conteúdo obsoleto: retorna conteúdo em cache em caso de erros na origem (500/502/503/504)

## Design de Componente Modular

Este é uma peça de um ambiente de laboratório maior. Cada componente é autocontido e implantado independentemente:

- **Este componente** fornece a borda CDN (NGINX em VM Azure)
- **Outros componentes** fornecem a aplicação de origem, configuração do F5 XC, DNS, políticas WAF, etc.

O operador humano adiciona componentes um de cada vez. A documentação de cada componente é escrita para que um assistente de IA possa lê-la e implantar a infraestrutura de forma autônoma.

## Por Que NGINX

O NGINX foi selecionado como motor de simulação CDN porque:

1. **Produto F5** — A F5 adquiriu a NGINX Inc. em 2019; faz parte do portfólio F5
2. **Comprovado na indústria** — O Cloudflare executou todo o seu CDN em NGINX por mais de uma década antes de migrar para o Pingora; a Netflix usa NGINX para seu CDN Open Connect
3. **Processo único** — lida com terminação TLS e cache em um único binário, diferentemente do Varnish que requer um proxy TLS separado
4. **Implantação simples** — `apt install nginx` no Ubuntu 24.04, duas diretivas habilitam o cache
5. **Bem documentado** — documentação oficial extensa para cache de conteúdo e configuração de proxy reverso
