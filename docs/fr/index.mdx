## Ce que cela fournit

- **[Nœud de périphérie unique](01-overview/)** — Une VM Ubuntu 24.04 exécutant NGINX en tant que proxy inverse avec mise en cache, le nœud de périphérie CDN minimum viable
- **[Déploiement Terraform](03-deploy/)** — Infrastructure Azure complète en tant que code : VNet, PIP, NSG et VM avec provisionnement cloud-init
- **[Plus de 67 en-têtes de fournisseurs CDN](04-nginx-config/)** — Simule les en-têtes de requêtes Akamai, Cloudflare, CloudFront, Fastly et Azure Front Door, y compris True-Client-IP, CF-Connecting-IP, géolocalisation, détection d'appareil, empreintes JA3/JA4 et scores de bots
- **[Intégration avec l'origine](06-integrate/)** — Configurez le nœud de périphérie pour transmettre les défauts de cache vers un répartiteur de charge HTTP F5 XC en tant qu'origine
- **[Vérification du cache](05-verify/)** — Validez le comportement HIT/MISS avec les en-têtes X-Cache-Status et l'inspection du cache NGINX

## Référence rapide

Après le déploiement, le nœud de périphérie CDN expose :

| Point de terminaison | Méthode | Objectif |
|---|---|---|
| `http://<PUBLIC_IP>/health` | GET | Vérification de santé — renvoie un JSON avec l'état des composants et les profils de fournisseurs |
| `http://<PUBLIC_IP>/` | GET | Proxy CDN — met en cache et transmet les requêtes au serveur d'origine |
| `http://<PUBLIC_IP>/<any-path>` | GET | Proxy CDN — tous les chemins sont transmis via proxy avec mise en cache et en-têtes de fournisseurs |

En-têtes de réponse ajoutés à chaque réponse transmise par proxy :

| En-tête | Exemple | Objectif |
|---|---|---|
| `X-Cache-Status` | `HIT` ou `MISS` | Indique si la réponse a été servie depuis le cache |
| `X-CDN-Edge` | `cdn-simulator` | Identifie ce nœud de périphérie |
| `X-CDN-POP` | `SJC` | Code simulé de Point de Présence |
| `X-Served-By` | `cache-sjc3120-SJC` | Identifiant simulé du nœud de cache |
| `X-Request-ID` | UUID | Identifiant unique par requête |
