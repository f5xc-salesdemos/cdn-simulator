## Ce que cela fournit

- **[Nœud Edge unique](01-overview/)** — Une VM Ubuntu 24.04 exécutant NGINX en tant que reverse proxy avec cache, le CDN edge minimal viable
- **[Déploiement Terraform](03-deploy/)** — Infrastructure Azure complète en tant que code : VNet, PIP, NSG et VM avec provisionnement cloud-init
- **[67+ en-têtes de fournisseurs CDN](04-nginx-config/)** — Simule les en-têtes de requête Akamai, Cloudflare, CloudFront, Fastly et Azure Front Door, incluant True-Client-IP, CF-Connecting-IP, géolocalisation, détection d'appareil, empreintes JA3/JA4 et scores de bot
- **[Intégration Origin](06-integrate/)** — Configurez le nœud edge pour transférer les défauts de cache vers un équilibreur de charge HTTP F5 XC en tant qu'origin
- **[Vérification du cache](05-verify/)** — Validez le comportement HIT/MISS avec les en-têtes X-Cache-Status et l'inspection du cache NGINX

## Référence rapide

Après le déploiement, le nœud CDN edge expose :

| Point de terminaison | Méthode | Objectif |
|---|---|---|
| `http://<PUBLIC_IP>/health` | GET | Vérification de santé — retourne un JSON avec l'état des composants et les profils de fournisseurs |
| `http://<PUBLIC_IP>/` | GET | Proxy CDN — met en cache et transfère les requêtes vers le serveur origin |
| `http://<PUBLIC_IP>/<any-path>` | GET | Proxy CDN — tous les chemins sont proxifiés avec mise en cache et en-têtes de fournisseurs |

En-têtes de réponse ajoutés à chaque réponse proxifiée :

| En-tête | Exemple | Objectif |
|---|---|---|
| `X-Cache-Status` | `HIT` ou `MISS` | Indique si la réponse a été servie depuis le cache |
| `X-CDN-Edge` | `cdn-simulator` | Identifie ce nœud edge |
| `X-CDN-POP` | `SJC` | Code de Point de Présence simulé |
| `X-Served-By` | `cache-sjc3120-SJC` | Identifiant de nœud de cache simulé |
| `X-Request-ID` | UUID | Identifiant unique par requête |
