Le provisionnement cloud-init déploie une configuration de périphérie CDN NGINX entièrement optimisée pour la performance. Cette page documente chaque couche de configuration, de l'ajustement du noyau au comportement du cache en passant par l'injection d'en-têtes de fournisseurs. Tous les paramètres ont été vérifiés lors d'un test de charge continu de 48 heures atteignant un pic de 172 540 req/s.

## Fichiers de Configuration

| Fichier | Objectif |
|---|---|
| `/etc/sysctl.d/99-cdn-tuning.conf` | Ajustement réseau du noyau Linux |
| `/etc/systemd/system/nginx.service.d/override.conf` | Limites de descripteurs de fichiers pour NGINX |

## Configuration Principale NGINX

### Workers et Connexions

```nginx
worker_processes auto;           # 4 workers on D4s_v5 (1 per vCPU)
worker_rlimit_nofile 65535;      # Per-worker file descriptor limit

events {
    use epoll;                   # Linux-optimized event model
    worker_connections 8192;     # 4 workers x 8192 = 32,768 max concurrent
    multi_accept on;             # Accept all pending connections per event loop
    accept_mutex off;            # Not needed with epoll + reuseport
}
```

Le remplacement systemd à `/etc/systemd/system/nginx.service.d/override.conf` définit `LimitNOFILE=65535` pour correspondre.

### Buffers Proxy

```nginx
proxy_buffering on;
proxy_buffer_size 16k;           # Header buffer (handles large CDN headers)
proxy_buffers 64 16k;            # 1 MB per connection (64 x 16k)
proxy_busy_buffers_size 256k;    # Can send 256k to client while still reading
```

La taille du busy buffer de 256k est critique — elle doit dépasser la plus grande réponse mise en cache (Juice Shop fait 75 Ko). Le paramètre original de 64k provoquait une sérialisation sous charge.

### Compression Gzip

```nginx
gzip on;
gzip_comp_level 4;               # Balance: ~85% of max compression at ~40% CPU
gzip_min_length 256;             # Skip tiny responses (gzip header overhead)
gzip_vary on;                    # Vary: Accept-Encoding for correct caching
gzip_proxied any;                # Compress all proxied responses
gzip_types text/plain text/css text/javascript text/xml
           application/json application/javascript application/xml
           application/xml+rss application/atom+xml
           application/ld+json application/manifest+json
           image/svg+xml;
```

### Cache de Fichiers Ouverts

```nginx
open_file_cache max=200000 inactive=20s;
open_file_cache_valid 30s;
open_file_cache_min_uses 2;
open_file_cache_errors on;
```

Met en cache les descripteurs de fichiers et les métadonnées des objets en cache, éliminant les appels système `stat()` et `open()` sur les fichiers fréquemment accédés.

### Keepalive Client

```nginx
keepalive_timeout 65;
keepalive_requests 100000;       # Was 1000 — caused connection recycling at 90K req/s
```

À 90K req/s avec `keepalive_requests 1000`, les connexions étaient recyclées toutes les 11 secondes, générant des sockets TIME_WAIT. L'augmentation à 100 000 a complètement éliminé ce problème.

### Journalisation des Accès

```nginx
log_format cdn '$remote_addr [$time_local] "$request" $status $body_bytes_sent $upstream_cache_status $request_time';
access_log /var/log/nginx/access.log cdn buffer=256k flush=5s;
```

Le format de log `cdn` inclut `$upstream_cache_status` (HIT/MISS) et `$request_time` pour l'analyse de latence. La journalisation bufferisée (256k/flush 5s) réduit la surcharge d'E/S sous forte charge.

## Keepalive Upstream

```nginx
upstream origin_backend {
    server 20.12.78.159:80;
    keepalive 256;               # Persistent connections per worker to origin
    keepalive_timeout 60s;
    keepalive_requests 1000;
}
```

Avec 4 workers x 256 keepalive = 1 024 connexions persistantes vers l'origine. Combiné avec `proxy_http_version 1.1` et `proxy_set_header Connection ""`, cela élimine la surcharge de l'établissement de connexion TCP à chaque cache miss.

## Configuration du Cache

### Chemin du Cache

```nginx
proxy_cache_path /var/cache/nginx/cdn
                 levels=1:2
                 keys_zone=cdn_cache:32m
                 max_size=25g
                 inactive=24h
                 use_temp_path=off;
```

| Paramètre | Valeur | Objectif |
|---|---|---|
| `keys_zone=cdn_cache:32m` | 32 Mo de mémoire partagée | Stocke environ 256 000 clés de cache et métadonnées |
| `max_size=25g` | Limite disque de 25 Go | Utilise la majeure partie du disque OS de 30 Go ; éviction LRU à la limite |
| `inactive=24h` | Timeout d'inactivité de 24 heures | Le contenu non accédé pendant 24h est évincé (différent du TTL) |
| `use_temp_path=off` | Écriture directe dans le répertoire de cache | Évite les déplacements de fichiers entre systèmes de fichiers |

### Comportement du Cache

```nginx
proxy_cache cdn_cache;
proxy_cache_valid 200 301 302 4h;
proxy_cache_valid 404 1m;
proxy_cache_key "$scheme$host$request_uri";
proxy_cache_lock on;
proxy_cache_lock_age 3s;
proxy_cache_lock_timeout 3s;
proxy_cache_background_update on;
proxy_cache_use_stale updating error timeout http_500 http_502 http_503 http_504;
proxy_ignore_headers Set-Cookie Cache-Control Expires Vary;

proxy_hide_header X-Cache-Status;
proxy_hide_header Vary;
```

| Directive | Valeur | Objectif |
|---|---|---|
| `proxy_cache_valid 200 301 302 4h` | TTL de 4 heures | Contenu en cache valide pendant 4 heures (augmenté pour réduire les baisses de débit lors des vagues de rafraîchissement) |
| `proxy_cache_lock on` | Prévention de l'effet de troupeau | Seule 1 requête va à l'origine par URL non mise en cache ; les autres attendent le remplissage du cache |
| `proxy_cache_lock_age 3s` | Timeout du verrou | Si la première requête n'est pas terminée en 3s, une autre est autorisée à passer |
| `proxy_cache_background_update on` | Rafraîchissement à latence zéro | Sert le contenu périmé immédiatement tout en le rafraîchissant en arrière-plan |
| `proxy_cache_use_stale` | Résilience | Sert le contenu périmé lors d'erreurs d'origine (500/502/503/504) ou pendant la mise à jour |
| `proxy_ignore_headers` | Forcer la mise en cache | Ignore les en-têtes Set-Cookie, Cache-Control, Expires, Vary de l'origine — le CDN décide du TTL et du comportement Vary (Juice Shop envoie `max-age=0` et des en-têtes `Vary` triples qui empêchaient une mise en cache efficace) |
| `proxy_hide_header X-Cache-Status` | Supprimer l'en-tête d'origine | L'NGINX d'origine ajoute son propre X-Cache-Status — le supprimer pour que seul le statut de cache du CDN soit visible |
| `proxy_hide_header Vary` | Prévenir la fragmentation du cache | L'origine envoie plusieurs en-têtes `Vary: Accept-Encoding`. Les supprimer empêche la fragmentation des clés de cache à travers les permutations `Accept-Encoding`. Le `gzip_vary on` de NGINX ajoute automatiquement le bon en-tête `Vary` unique |

### Timeouts Proxy

```nginx
proxy_read_timeout 30s;
proxy_connect_timeout 10s;
proxy_send_timeout 15s;
```

Donne à l'origine plus de temps de réponse sous charge tout en échouant rapidement sur les problèmes de connexion.

### Bloc Server

```nginx
server {
    listen 80 reuseport;        # Kernel distributes connections across all 4 workers
    server_name _;
}
```

`reuseport` active `SO_REUSEPORT` — le noyau distribue les connexions entrantes directement aux processus workers, éliminant la contention du mutex d'acceptation.

## Détection d'Appareil

Le simulateur détecte le type d'appareil à partir de l'en-tête `User-Agent` en utilisant les directives `map` de NGINX :

- **Mobile** : Correspond à iPhone, Android (non-tablette), iPod, BlackBerry, Opera Mini, IEMobile
- **Tablette** : Correspond à iPad, tablette Android, Kindle, PlayBook
- **Bureau** : Par défaut lorsque ni mobile ni tablette ne correspond

Le type d'appareil est reflété dans :
- `CloudFront-Is-Desktop-Viewer` / `CloudFront-Is-Mobile-Viewer` / `CloudFront-Is-Tablet-Viewer`
- `X-Akamai-Device-Characteristics` (champs `is_mobile`, `is_tablet`, `is_wireless_device`)

## Opérations

### Changer le Serveur d'Origine

```bash
ssh azureuser@<PUBLIC_IP>

# Update upstream server
sudo sed -i 's|server .*;|server NEW_HOST:80;|' /etc/nginx/conf.d/cdn-edge.conf

# Clear cache and reload
sudo rm -rf /var/cache/nginx/cdn/*
sudo nginx -t && sudo systemctl reload nginx
```

Ou mettez à jour `origin_host` dans `terraform.tfvars` et exécutez `terraform apply` pour reprovisionner.

### Vider le Cache

```bash
ssh azureuser@<PUBLIC_IP>
sudo rm -rf /var/cache/nginx/cdn/*
sudo systemctl reload nginx
```

### Vérifier les Statistiques du Cache

```bash
# Object count and disk usage
sudo find /var/cache/nginx/cdn -type f | wc -l
sudo du -sh /var/cache/nginx/cdn

# Recent access log with cache status
tail -20 /var/log/nginx/access.log
```

### Surveillance Sous Charge

```bash
# Real-time connections and socket states
ss -s

# NGINX worker CPU usage
top -bn1 | grep nginx

# Upstream keepalive connections
ss -tn state established dst <ORIGIN_IP> | wc -l

# TIME_WAIT socket count
ss -tn state time-wait | wc -l
```
