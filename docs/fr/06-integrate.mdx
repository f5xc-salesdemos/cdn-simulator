Cette page couvre deux étapes d'intégration :

1. **Intégration directe** — Le CDN edge transmet au serveur d'origine directement (tests de référence)
2. **Insertion F5 XC** — Le répartiteur de charge HTTP F5 XC est inséré entre le CDN et l'origine (démonstration de sécurité)

Commencez par l'intégration directe pour établir la ligne de référence, puis insérez F5 XC lorsque vous êtes prêt.

## Référence du serveur d'origine

Le composant de laboratoire [origin-server](https://f5xc-salesdemos.github.io/origin-server/) fournit des applications web vulnérables pour les tests de sécurité.

| Propriété | Valeur |
|---|---|
| Documentation | [f5xc-salesdemos.github.io/origin-server](https://f5xc-salesdemos.github.io/origin-server/) |
| Dépôt | [github.com/f5xc-salesdemos/origin-server](https://github.com/f5xc-salesdemos/origin-server) |
| Port par défaut | 80 |
| Vérification de santé | `GET /health` |

### Applications disponibles

Récupérez le catalogue d'applications actuel à partir du manifeste publié du serveur d'origine :

```bash
MANIFEST_URL=$(curl -sf https://api.github.com/repos/f5xc-salesdemos/origin-server/releases/latest \
  | python3 -c "import sys,json; assets=json.load(sys.stdin).get('assets',[]); print(next((a['browser_download_url'] for a in assets if a['name']=='manifest.json'),''))")
curl -sf "$MANIFEST_URL" | python3 -m json.tool
```

Si aucune release n'existe encore, utilisez directement la source du dépôt :

```bash
curl -sf https://raw.githubusercontent.com/f5xc-salesdemos/origin-server/main/manifest.json | python3 -m json.tool
```

Le manifeste répertorie tous les chemins d'applications, les vérifications de santé, les images de conteneurs et les correspondances avec les fonctionnalités de démonstration.

## Étape 1 : Intégration directe (ligne de référence)

```
┌──────────┐     ┌──────────────────────┐     ┌─────────────────────┐
│  Client   │────▶│  CDN Edge (NGINX)    │────▶│  Origin Server      │
│           │     │  20.65.90.112        │     │  20.12.78.159       │
└──────────┘     │  67+ CDN headers     │     │  Juice Shop, DVWA,  │
                 │  Disk cache           │     │  VAmPI, httpbin,    │
                 └──────────────────────┘     │  whoami, CSD Demo   │
                                              └─────────────────────┘
```

### Configurer l'origine

Définissez l'adresse IP du serveur d'origine dans la configuration NGINX du CDN edge :

```bash
ssh azureuser@<CDN_EDGE_IP>
sudo sed -i 's|proxy_pass .*;|proxy_pass http://<ORIGIN_IP>;|' /etc/nginx/conf.d/cdn-edge.conf
sudo rm -rf /var/cache/nginx/cdn/*
sudo nginx -t && sudo systemctl reload nginx
```

Ou définissez-la au moment du déploiement Terraform via `terraform.tfvars` :

```hcl
origin_server = "http://<ORIGIN_IP>"
```

### Vérifier toutes les applications

Testez chaque application d'origine à travers le CDN :

```bash
CDN=<CDN_EDGE_IP>

# Health check (CDN local)
curl -sf "http://$CDN/health" | python3 -m json.tool

# Landing page
curl -sf -o /dev/null -w "/ : HTTP %{http_code}\n" "http://$CDN/"

# Juice Shop
curl -sf -o /dev/null -w "/juice-shop/ : HTTP %{http_code}\n" "http://$CDN/juice-shop/"

# DVWA (follows redirect to login)
curl -sf -o /dev/null -w "/dvwa/ : HTTP %{http_code}\n" -L "http://$CDN/dvwa/"

# VAmPI API
curl -sf "http://$CDN/vampi/users/v1" | python3 -m json.tool | head -5

# httpbin headers (shows CDN headers in JSON)
curl -sf "http://$CDN/httpbin/headers" | python3 -m json.tool | head -10

# whoami (shows ALL headers the origin receives)
curl -sf "http://$CDN/whoami/"

# CSD Demo
curl -sf -o /dev/null -w "/csd-demo/ : HTTP %{http_code}\n" "http://$CDN/csd-demo/"
```

Tous les chemins doivent retourner HTTP 200 (DVWA retourne 302 puis 200 lors du suivi de redirection).

### Vérifier les en-têtes CDN au niveau de l'origine

Le point de terminaison `/whoami/` affiche chaque en-tête que l'origine reçoit. Lorsqu'il est accédé via le CDN, il affiche les 67+ en-têtes de fournisseurs :

```bash
curl -sf "http://$CDN/whoami/"
```

Vérifiez que ces en-têtes clés sont présents :

| Fournisseur | En-tête | Valeur attendue |
|---|---|---|
| Standard | `X-Forwarded-For` | `<your_ip>, <cdn_edge_ip>` |
| Akamai | `True-Client-Ip` | `<your_ip>` |
| Cloudflare | `Cf-Connecting-Ip` | `<your_ip>` |
| CloudFront | `Cloudfront-Viewer-Country` | `US` |
| Fastly | `Fastly-Client-Ip` | `<your_ip>` |
| Azure FD | `X-Azure-Clientip` | `<your_ip>` |

### Corrélation croisée des journaux

Comparez les journaux d'accès sur les deux serveurs pour vérifier le flux de trafic :

```bash
# CDN edge log — shows your client IP as source
ssh azureuser@<CDN_EDGE_IP> "sudo tail -5 /var/log/nginx/access.log"

# Origin log — shows CDN edge IP as source
ssh azureuser@<ORIGIN_IP> "sudo tail -5 /var/log/nginx/access.log"
```

Le journal de l'origine doit afficher l'adresse IP du CDN edge comme client connecté, tandis que l'adresse IP réelle du client est transportée dans `X-Forwarded-For` et les en-têtes spécifiques aux fournisseurs.

### Comportement du cache

```bash
# First request — MISS (fetched from origin)
curl -s -I "http://$CDN/whoami/" | grep X-Cache-Status

# Second request — HIT (served from CDN cache)
curl -s -I "http://$CDN/whoami/" | grep X-Cache-Status
```

## Étape 2 : Insertion F5 XC (démonstration de sécurité)

Après les tests de référence, insérez un répartiteur de charge HTTP F5 XC entre le CDN et l'origine :

```
┌──────────┐     ┌────────────────┐     ┌──────────────────┐     ┌─────────────────┐
│  Client   │────▶│  CDN Edge      │────▶│  F5 XC HTTP LB   │────▶│  Origin Server  │
│           │     │  (NGINX)       │     │  WAF + Bot + API  │     │                 │
└──────────┘     └────────────────┘     └──────────────────┘     └─────────────────┘
```

1. **Créez le répartiteur de charge HTTP F5 XC** avec le serveur d'origine dans son pool d'origines
2. **Mettez à jour le CDN edge** pour pointer vers le VIP F5 XC au lieu de l'origine directement :

```bash
ssh azureuser@<CDN_EDGE_IP>
sudo sed -i 's|proxy_pass .*;|proxy_pass https://<F5XC_LB_VIP>;|' /etc/nginx/conf.d/cdn-edge.conf
sudo rm -rf /var/cache/nginx/cdn/*
sudo nginx -t && sudo systemctl reload nginx
```

3. **Vérifiez l'application du WAF** à travers la chaîne complète :

```bash
# SQL injection through CDN → F5 XC WAF should block
curl -I "http://$CDN/dvwa/vulnerabilities/sqli/?id=%27+OR+1%3D1--"

# Normal request should pass
curl -sf -o /dev/null -w "%{http_code}" "http://$CDN/juice-shop/"
```

4. **Configurez l'en-tête d'IP client de confiance** dans F5 XC pour lire l'adresse IP réelle du client à partir des en-têtes CDN (par exemple, `True-Client-IP` pour la simulation Akamai, `CF-Connecting-IP` pour la simulation Cloudflare)

## Architecture multi-composants

| Composant | Dépôt | Objectif |
|---|---|---|
| **CDN Edge** (celui-ci) | [cdn-simulator](https://github.com/f5xc-salesdemos/cdn-simulator) | Mise en cache, en-têtes de fournisseurs |
| **Origin Server** | [origin-server](https://github.com/f5xc-salesdemos/origin-server) | Applications web vulnérables |
| **F5 XC Config** | Divers (waf, api-protection, bot-*, etc.) | Politiques de sécurité |

Chaque composant publie une documentation que les assistants IA lisent pour déployer l'infrastructure. Le serveur d'origine publie un [manifeste de points de terminaison](https://github.com/f5xc-salesdemos/origin-server/releases/latest) en tant qu'artefact de Release GitHub répertoriant tous les chemins d'applications et les vérifications de santé.
